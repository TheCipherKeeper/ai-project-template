# AGENTS.md — правила методологии

Этот репозиторий содержит правила и скелеты agent-first разработки для одного
человека и программных агентов. Начальная точка — `docs/START.md`, выполнение
задачи — `docs/WORKFLOW.md`, роутер — `docs/INDEX.md`.

## Авторитет

Порядок приоритета: этот `AGENTS.md` → `docs/` → `skeletons/` → код verifier.
Противоречие внутри одного уровня является дефектом. Один факт хранится в одном
месте; документы ссылаются на него, а не пересказывают.

## Работа человека и агентов

- Человек пишет и упорядочивает задачи, отвечает на продуктовые вопросы и видит
  результат в test.
- Человек единолично определяет необходимые репозитории, их количество,
  назначение и высокоуровневые границы системы. Агенты не создают и не удаляют
  репозитории и не меняют их назначение без явной задачи человека.
- Агенты выполняют первый `[ ] ready`, но не додумывают отсутствующий intent.
- Одновременно выполняется одна задача; lock-PR и статус `[~]` запрещены.
- Обычная функциональность получает review независимым вызовом. Специальные
  проверки включаются только по триггерам из `docs/REFERENCE.md`.
- Число исправлений ограничено правилами `docs/WORKFLOW.md`.
- Агент может повысить риск, но не может понизить его, принять остаточный риск
  или расширить собственную автономность.

## Ветвление и выпуски

- `main` — единственная интеграционная ветка.
- Работа — `feat/TASK-NNNN-<slug>` и PR с обязательными checks.
- Прямой коммит в `main`, ветки `dev` и release-ветки запрещены.
- Merge выполняется squash после зелёного гейта.
- Если человек явно не запретил публикацию, агент сам доводит изменение до
  удалённого `main`: push feature-ветки → PR → обязательные checks → squash
  merge → синхронизация локального `main`. Остановка на локальном commit не
  считается завершением. Канонический порядок — `docs/WORKFLOW.md`.
- Test deployment идентифицируется commit SHA/digest. Стабильный `vX.Y.Z` и
  необязательный `vX.Y.Z-rc.N` создаются отдельной задачей.

## Что можно менять

- `docs/`, корневые `README.md` и `AGENTS.md`;
- `skeletons/{hub,service,interface,stub}/`;
- verifier и репозиторные инструменты на Python;
- JSON-схемы и CI, необходимые для гейта.

## Ограничения

- Не создавать репозиторные `*.sh` и `*.ps1`; инструменты пишутся на Python и
  запускаются через `uv`.
- Не хранить в `skeletons/` код приложения и lock-файлы.
- Не класть сервисные или хабовые рабочие артефакты в корень методологии.
- Не трогать `.env`, lock-файлы и секреты без явного разрешения.
- Не выдавать placeholder/TODO за реализацию.
- Не вводить обязательную распределённость и не предлагать новый репозиторий как
  скрытую часть реализации.

## Проверка

Перед коммитом:

```bash
uv run tools/verify/verify.py --report verification.json
```

Гейт проверяет обязательные файлы, ссылки, схемы, запрещённые артефакты и
структурные правила скелетов. Его недоступность не считается успехом.

## Коммиты и язык

Conventional Commits; допустимые scope: `docs`, `skeletons`, `verify`.
Документация — русская, идентификаторы и summary коммита — английские.
Несовместимость отмечается `BREAKING CHANGE:` в теле коммита.
