# AGENTS.md — правила работы в репозитории

Этот файл — для людей и AI-агентов, которые работают с кодом. Здесь только
правила. Архитектура — в `docs/ARCHITECTURE.md`, раскладка — в `docs/LAYOUT.md`,
деплой — в `docs/DEPLOYMENT.md`, контракты модулей — в `docs/specs/`, детали
стека — в `docs/STACKS.md`.

> Это шаблон **одного микросервиса**. Один репо = один сервис. Внутри сервис
> может быть workspace'ом из нескольких модулей («монорепо в рамках сервиса»),
> у каждого модуля своя спека. Сервис реализуется **на одном из стеков**:
> Python, Go, Rust, TypeScript (бэкенд). Правило: один сервис — один язык.
> Сервис — клиент **брокера** (одного на систему; Kafka/Redpanda/NATS) и
> деплоится **контейнером** со своим `Dockerfile`.
>
> **Программа целиком** — несколько сервисов + хаб + системный compose +
> COMPOSITION/CONVENTIONS/ADR — живёт в хабе и в отдельных репо сервисов.
> Этот репо описывает один сервис. Кросс-сервисные контракты, системная
> топология и ADR — в хабе (см. *ADR* ниже).

## Документация

Документы в порядке убывания приоритета:

1. Хаб (если есть) — `COMPOSITION.md` (состав), `CONVENTIONS.md` (event envelope,
   кросс-сервисные конвенции), `adr/` (архитектурные решения).
2. `AGENTS.md` (этот файл) — правила работы.
3. `docs/ARCHITECTURE.md` — архитектура сервиса.
4. `docs/BACKLOG.md` — очередь задач.
5. `docs/VERIFICATION.md` — verification gate (соответствие канону на коммит).
6. `docs/specs/*.md` — контракты модулей сервиса.
7. Код — реализация.

`docs/INDEX.md` — карта по всем документам. Начни с него.

Если документы противоречат друг другу — побеждает старший. Расхождение —
повод для ADR.

## Модель ветвления

```
main ← merge только из dev
dev  ← PR из feature-веток
feat/<задача> ← от dev, удаляется после merge
```

- `main` — стабильная. Вливается только из `dev`.
- `dev` — разработческая. Вливается из feature-веток через PR.
- Прямой коммит в `main` или `dev` — запрещён. Только через feature-ветку + PR.

## Рабочий цикл

### Важно!
Бэклог — это строгая последовательная очередь (как миграции). Запрещено выполнять задачи параллельно или перепрыгивать через невыполненный пункт. Задача $N+1$ может быть начата только после полного завершения задачи $N$.

1. Прочитать `docs/BACKLOG.md` — взять самый первый невыполненный `[ ]` пункт.
   Тег задачи указывает модуль, которого касается задача.
2. Прочитать `docs/INDEX.md` → `docs/ARCHITECTURE.md` → спек нужного модуля
   (возможно нескольких необходимых).
3. Создать ветку от dev: `git checkout dev && git pull && git checkout -b feat/<задача>`.
4. Изменить код. Написать тесты по пунктам «Что есть» в спеке.
5. Проверить — все три группы команд должны пройти (команды под свой стек —
   из таблицы ниже): **lint**, **test**, **build**. Если любая fail — чинить, повторить.
6. Обновить спек: перенести реализованное из «Что TODO» в «Что есть».
7. Обновить BACKLOG: поставить `[x]` на выполненный пункт.
8. Закоммитить (Conventional Commits, см. ниже).
9. Запушить, открыть PR в `dev`.
10. После merge — ветка удаляется.

## Команды проверки по стеку

Стек выбирается один раз для всего репо. Подставь свой набор в таблицу и держи
его единым. Полная конфигурация toolchain'а — в `docs/STACKS.md`.

| Стек | lint | test | build |
|---|---|---|---|
| **Python** | `ruff format --check . && ruff check .` | `pytest` | `uv build` |
| **Go** | `gofmt -l . && go vet ./...` | `go test ./...` | `go build -o bin/<service> ./cmd/<service>` |
| **Rust** | `cargo fmt --check && cargo clippy -- -D warnings` | `cargo test` | `cargo build --release` |
| **TypeScript** | `pnpm lint && tsc --noEmit` | `pnpm test` | `pnpm build` |

> Go: `gofmt -l .` должен вывести пустоту — это и есть «fmt --check».
> Python: опционально `mypy`/`pyright` для типизации.
> TypeScript: runtime — Node (по умолчанию); Deno/Bun допустимы, но зафиксируй в `docs/STACKS.md`.

## Интеграция с брокером

- Сервис — **клиент брокера** (одного на систему; Kafka/Redpanda/NATS).
  Брокер как зависимость описывается в `docker-compose.yml` и `docs/ARCHITECTURE.md`.
- Топики сервиса (publish/consume) — в `docs/ARCHITECTURE.md` → *Брокер*.
- Формат сообщений (event envelope) — из хаба `CONVENTIONS.md` (или
  `docs/CONVENTIONS.md`, если хаба нет). Не изобретать свой формат в спеках.
- Прямой связности с другими сервисами в обход брокера здесь нет — это
  выход за границу одного сервиса и решается в хабе/ADR.

## Деплой

- Сервис деплоится **контейнером** со своим `Dockerfile` в корне репо.
- `docker-compose.yml` в корне — **только для локальной разработки**: брокер +
  этот сервис. Системный compose (все сервисы вместе) — в хабе/деплой-репо, не здесь.
- Сборка образа — из `Dockerfile`; команды сборки — в рамках `build` стека.
- Переменные — `.env` (не коммитится) / `.env.example` (шаблон).

## Верификация (verification gate)

На каждый коммит автоматически проверяется соответствие канону. Идея: на
изменение в узле графа проверять все инцидентные рёбра — **вверх**
(соответствие авторитету) и **вниз** (соответствие детей). Этот репо —
листовой сервис, для него активно ребро вверх: `хаб → сервис`. Полная схема,
версионирование контрактов, conformance-vs-behavioral и invariant-чеклист —
в `docs/VERIFICATION.md`.

- Правила гейт читает из **авторитета** (хаб/шаблон), не из копии в репо —
  иначе рассинхронизация копии с источником не ловится.
- Сервис **пинит версию контрактов хаба** (`CONVENTIONS@v<N>`); гейт проверяет
  против пина, не HEAD.
- Детерминированные проверки блокируют (CI: lint/test/build + rule-инварианты);
  агентские conformance-проверки — консультативны и **не** блокируют.

## Что можно

- Писать код в модулях сервиса (workspace) и (опц.) `shared/`.
- Менять конфигурацию сборки/манифесты с обоснованием.
- Менять `Dockerfile`, корневой `docker-compose.yml` (dev: брокер + сервис),
  `.env.example` с обоснованием.
- Обновлять `docs/` и спеки.
- Создавать feature-ветки, коммитить, пушить, открывать PR в `dev`.
- Заводить новые модули в workspace'е (с соответствующей спекой).

## Что нельзя

- Коммитить напрямую в `main` или `dev`.
- Смешивать стеки (один сервис — один язык).
- Вводить системный multi-service compose или кросс-сервисные контракты в этом
  репо — это зона хаба.
- Создавать ADR вне отведённого места (см. *ADR* ниже).
- Добавлять зависимости (включая образы в compose) без обоснования.
- Выдавать stub за реальную реализацию. Честно помечать: что placeholder, что TODO.
- Трогать lock-файлы (`Cargo.lock`, `go.sum`, `pnpm-lock.yaml`, `uv.lock`),
  конфиги окружения (`.env`), артефакты сборки без одобрения.

## Коммиты

Conventional Commits. Полностью на английском (или на языке проекта — задай в *Язык*).

```
feat(host): add zfs-snapshot fs probe
fix(core): reject path traversal in HostBridge
docs: update ARCHITECTURE.md with module matrix
refactor(transport): extract envelope signing
chore(deploy): pin redpanda image in compose
```

Scope — имя модуля или `deploy`/`docs`. Breaking changes —
`BREAKING CHANGE:` в теле.

## ADR

Архитектурные решения фиксируются в ADR. Для cybercity ADR по умолчанию живут
в **хабе** (`cybercity/adr/`), а из `docs/ARCHITECTURE.md` ссылаются на них.

> **Заметка про standalone.** Если сервис — отдельный проект без хаба, ADR
> живут в `docs/adr/` этого репо (формат — `docs/adr/0001-record-architecture-decisions.md`).
> Старые ADR помечай `Superseded`, не удаляй.
