# Фаза 70 — Выпустить стабильную версию (тег)

**Цель:** зафиксировать стабильную версию сервиса тегом `vX.Y.Z` на `main`.

**Когда:** `main` достиг состояния, которое хочется назвать релизом (набор
слитых feature-веток, прошёл гейт, локально проверено).

**Что нужно до:** `main` чистый, всё слито через PR, гейт зелёный
(`docs/guide/40-verify.md`), образ собирается (`docs/guide/50-deploy.md`).

> Стабильные версии — **тегами, не ветками**. Release-ветки не заводятся:
> точка релиза — тег на `main`, источник для отката/чекаута — он же.

## Шаги

1. **Обновить `main` и убедиться, что он чистый:**
   ```bash
   git checkout main && git pull
   git status -sb        # working tree clean
   ```
2. **Прогнать финальную проверку** — `docs/guide/40-verify.md`
   (lint/test/build) + `docker compose build` (`docs/refs/DEPLOYMENT.md`).
3. **Определить версию** (semver `vX.Y.Z`):
   - `PATCH` (Z) — багфикс, без изменения контрактов.
   - `MINOR` (Y) — новая функциональность, обратно совместимая.
   - `MAJOR` (X) — breaking change (в т.ч. бамп пина `CONVENTIONS@v<N>`).
4. **Поставить annotated-тег** на `main`:
   ```bash
   git tag -a v0.1.0 -m "v0.1.0: <краткое описание релиза>"
   ```
5. **Запушить тег:**
   ```bash
   git push origin v0.1.0
   ```
6. **(Опц.) release notes** — список слитых PR / пункт BACKLOG с момента
   прошлого тега. Можно в описании тега или в GitHub Release.

## Канон (где правда)

- `AGENTS.md` → *Модель ветвления* — теги вместо release-веток.
- `docs/guide/40-verify.md` — что должно быть зелёным перед тегом.
- `docs/refs/VERIFICATION.md` — пин `CONVENTIONS@v<N>` (MAJOR-бамп при breaking).

## Что после

- Тег — точка релиза; хаб/деплой-репо ссылаются на собранный образ этого тега.
- Следующая разработка — новые `feat/<задача>` от `main` (фаза 30).
- Hotfix на прошлый релиз — `feat/<задача>` от тега, PR в `main`, новый тег
  (PATCH/MINOR). Отдельной ветки релиза не нужно.
