# Этап 70 — Выпустить версию

**Цель:** отметить версию сервиса тегом в `main`. Используются кандидаты на
выпуск и стабильные версии.

**Когда:**
- **Кандидат** выпускается после успешной проверки `main` и сборки образа. В
  автономном цикле кандидат создаётся **автоматически после слияния** с `main`
  (развёртывание в испытательной среде — первая
  точка человека; баг → `<hub>/BACKLOG`; модель — `docs/refs/PIPELINE.md`).
- **Стабильная версия** выпускается после проверки накопленных кандидатов.

**Что требуется до начала:** рабочее дерево `main` не содержит изменений, все
запросы на слияние приняты, проверки завершены успешно,
(`docs/guide/40-verify.md`), образ собирается (`docs/guide/50-deploy.md`).

> Версии отмечаются тегами, а не ветками. Ветки выпусков не создаются. Тег в
> `main` является точкой выпуска и восстановления. Суффикс `-rc.N` обозначает
> кандидата; отсутствие суффикса — стабильную версию. Другие суффиксы
> предварительных версий не применяются.

## Уровни версий

- **Кандидат** — `vX.Y.Z-rc.N`. В
  GitHub Release — галочка *Set as a pre-release*. По semver ниже чистого
  `vX.Y.Z`, то есть формально «ещё не стабильная».
- **Стабильная версия** — `vX.Y.Z` без суффикса. GitHub Release отмечается
  флажком *Set as the latest release*.

## Выпуск кандидата

1. **Обновить `main` и убедиться, что он чистый:**
   ```bash
   git checkout main && git pull
   git status -sb        # working tree clean
   ```
2. **Прогнать проверку** — `docs/guide/40-verify.md` (lint/test/build)
   + `docker compose build` (`docs/refs/DEPLOYMENT.md`).
3. **Поставить annotated-тег с суффиксом** на `main`:
   ```bash
   git tag -a v0.2.0-rc.1 -m "v0.2.0-rc.1: <краткое описание кандидата>"
   ```
4. **Запушить тег:**
   ```bash
   git push origin v0.2.0-rc.1
   ```
5. **GitHub Release** на теге — поставить *Set as a pre-release*.

## Выпуск стабильной версии

1. **Обновить `main` и убедиться, что он чистый:**
   ```bash
   git checkout main && git pull
   git status -sb        # working tree clean
   ```
2. **Прогнать финальную проверку** — `docs/guide/40-verify.md`
   (lint/test/build) + `docker compose build` (`docs/refs/DEPLOYMENT.md`).
3. **Определить версию** (semver `vX.Y.Z`):
   - `PATCH` (Z) — багфикс, без изменения контрактов.
   - `MINOR` (Y) — новая функциональность, обратно совместимая.
   - `MAJOR` (X) — breaking change (в т.ч. смена `CONVENTIONS`).
4. **Поставить annotated-тег без суффикса** на `main`:
   ```bash
   git tag -a v0.1.0 -m "v0.1.0: <краткое описание релиза>"
   ```
5. **Запушить тег:**
   ```bash
   git push origin v0.1.0
   ```
6. **GitHub Release** на теге — *Set as the latest release*, release notes:
   список слитых PR / пункт `<hub>/BACKLOG` с момента прошлого стабильного тега.

## Канон (где правда)

- `AGENTS.md` → *Модель ветвления* — теги вместо веток выпусков.
- `docs/guide/40-verify.md` — что должно быть зелёным перед тегом.
- `docs/refs/VERIFICATION.md` — модель контроля соответствия.
- `docs/refs/COMMUNICATION.md` — `CONVENTIONS` (MAJOR-бамп при breaking).

## Что после

- Тег — точка релиза; хаб ссылается на собранный образ стабильного тега
  (теги кандидатов используются для внутренних проверок).
- Следующая разработка — новые `feat/<задача>` от `main` (фаза 30).
- Срочное исправление прошлой стабильной версии выполняется в ветке
  `feat/<задача>` от соответствующего тега, затем создаётся запрос на слияние с `main` и
  новый тег (PATCH/MINOR). Отдельной ветки релиза не нужно.
