# Этап 10 — Описать архитектуру сервиса

**Цель:** заполнить `docs/ARCHITECTURE.md` — роль сервиса, модули, брокер,
топики, потоки данных, доверительная граница.

**Когда:** после подготовки репозитория (этап 00), до написания кода и
спецификаций модулей.

**Что требуется до начала:** завершён этап 00, выбран стек и создана структура
каталогов.

## Шаги

Пройди по секциям `docs/ARCHITECTURE.md` (скелет-комментарии подсказывают, что
писать в каждой):

1. **Что это** — 1–2 предложения: какой сервис, какую роль в программе играет.
   Ссылка на хаб `COMPOSITION.md` для системного контекста.
2. **Что делает** — нумерованный список ключевых функций (глагол → что/зачем).
3. **Чего не делает** — явные границы. Важно для агентов, чтобы не «помогали»
   там, где не надо.
4. **Модули** — таблица с названием и назначением каждого модуля, а также с
   публикуемыми и читаемыми темами. Зависимости между модулями задаются
   направленным ациклическим графом Mermaid.
5. **Брокер** — один на систему (Kafka / Redpanda / NATS), фиксируется в хабе
   (`CONVENTIONS`/`COMPOSITION`); сервис его **потребляет, не выбирает**. Адрес
   (локальная разработка) — из `docker-compose.yml`, сервис `broker`.
   - **Соглашения хаба:** `CONVENTIONS` содержит действующие требования, на
     соответствие которым проверяется сервис.
   - Топики сервиса: таблица (топик / направление publish|consume / назначение).
   - Формат сообщений берётся из `CONVENTIONS.md` хаба. Сервис не определяет
     собственный конверт сообщения.
6. **Потоки данных** — потоки этого сервиса через брокер (входящие/исходящие),
   mermaid-граф + шаги каждого потока.
7. **Доверительная граница** — где проходит граница доверия сервиса, что по
   какую сторону, гарантии, подпись/аутентификация. Для сервисов без границ —
   секцию убрать. Здесь же — **назначение gateway-роли** (см. ниже).
   - Если сервис выполняет роль **сервиса-шлюза** (см.
     `docs/refs/COMMUNICATION.md` → *Сервис-шлюз*), здесь документируются
     клиентские точки HTTP- и WebSocket-API: путь, метод, версия и формат
     ответа. Агент проверяет соответствие заявленных интерфейсом обращений этой
     таблице. В системе назначается не более одного сервиса-шлюза. Если
     пользовательских интерфейсов нет, сервис-шлюз не требуется.
   - Обычный сервис не предоставляет клиентский API. Его данные поступают в
     сервис-шлюз через темы брокера.
8. **Развёртывание и ссылки** — ссылка на `docs/guide/50-deploy.md` и
   `docs/refs/DEPLOYMENT.md`; ссылки на хаб `COMPOSITION.md`, `CONVENTIONS.md`,
   хаб `adr/`.

## Канон (где правда)

- `docs/ARCHITECTURE.md` — сам файл (скелет-комментарии).
- Хаб `COMPOSITION.md` (состав системы), `CONVENTIONS.md` (конверт сообщения).
- `docs/refs/VERIFICATION.md` — модель проверяемых связей и инварианты.
