# Этап 00 — Подготовить репозиторий сервиса

**Цель:** от пустого клона скелета к репо сервиса, готовому к разработке.

**Когда:** первый вход в репо — человеком или агентом.

**Что нужно до:** скопирован `skeletons/service/` из репо методологии в новый
репо сервиса (`docs/INDEX.md` → *Инстанциация*).

**Применимо:** сервис и gateway-сервис. Не применяется к хабу, интерфейсу и
автономному компоненту: для них точка входа — `docs/INDEX.md`.

**Результат:** сервис-репозиторий имеет выбранный стек, запускаемые команды
`lint / test / build`, начальную структуру и спеки модулей.

## Шаги

1. **Создать репо сервиса** из `skeletons/service/`: скопируй содержимое в новый
   репо, оторви историю:
   ```bash
   git checkout --orphan main && git add -A && git commit -m "init"
   ```
2. **Выбрать стек** (один на сервис; правило: один сервис — один язык):
   Python / Go / Rust / TypeScript.
   - Центральный `docs/refs/STACKS.md` не изменяй: он содержит все допустимые
     стеки. В `AGENTS.md` **сервис-репозитория** оставь одну строку выбранного
     стека и удали остальные строки таблицы → *Команды проверки*.
3. **Переименовать плейсхолдеры** `<service>` под имя сервиса: в
   `docker-compose.yml`, `Dockerfile` (когда заведёшь), манифесте
   (`pyproject.toml` / `go.mod` / `Cargo.toml` / `package.json`), `.env.example`.
4. **Создать каталоги модулей** по структуре выбранного стека
   (`docs/refs/LAYOUT.md`): Rust `src/`, Go `internal/`+`pkg/`,
   Python `src/<service>/`, TypeScript `src/`.
5. **Завести первые спеки:** удали `docs/specs/EXAMPLE.md`, заведи
   `docs/specs/<module>.md` по канону (`docs/refs/SPEC.md`). Подробно — этап 20.
6. **Выполнить статический анализ, тестирование и сборку** выбранного стека (команды —
   `docs/refs/STACKS.md`). Должны запускаться (пусть даже на пустом скелете).

## Канон (где правда)

- `docs/refs/STACKS.md` — набор инструментов, структура и команды стека.
- `docs/refs/LAYOUT.md` — раскладка каталогов сервиса.
- `docs/refs/SPEC.md` — структура спеки.

## Что после

- `docs/guide/10-architecture.md` — описать сервис (роль, модули, брокер, топики).
- Затем `docs/guide/20-define-module.md` — детально по каждому модулю.

Критерий завершения: плейсхолдеры заменены, структура выбранного стека создана,
а закреплённые в сервисном `AGENTS.md` команды `lint / test / build` проходят.
