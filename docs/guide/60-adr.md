# Этап 60 — Зафиксировать архитектурное решение

**Цель:** зафиксировать значимое архитектурное решение с обоснованием, чтобы
агентам и новым разработчикам было понятно, почему система устроена так.

**Когда:** спорное изменение, расхождение доков, архитектурный поворот,
откат решения без знания обоснования — иначе риск вернуть отбракованное.

**Что нужно до:** ничего.

## Когда ADR, а когда — просто правка

ADR нужен, если решение **значимое и с обоснованием**, которое будущий
читатель должен знать. Мелкие правки, переименования, локальные фиксы —
не ADR.

## Шаги

1. **Где живёт ADR:** в **хабе** `<hub>/adr/`. Из `docs/ARCHITECTURE.md`
   сервиса/интерфейса ссылаются на них (хаб — единый ADR-дом программы).
2. **Создать файл.** Скопируй `<hub>/adr/_TEMPLATE.md` → `000N-<slug>.md`
   (однострочная нумерация, `NNNN` → следующий номер).
3. **Заполнить** три секции:
   - **Контекст** — почему решение возникает: проблема, ограничения, варианты.
   - **Решение** — что решили, конкретно.
   - **Последствия** — что меняется: позитивное, негативное, риски, влияние дальше.
4. **Status:** `Proposed` → `Accepted`. Старые решения помечай `Superseded by
   ADR-NNNN`, **не удаляй** — история должна оставаться читаемой.
5. **Сослаться** на ADR из `docs/ARCHITECTURE.md` (в соответствующей секции).

## Канон (где правда)

- `<hub>/adr/_TEMPLATE.md` — шаблон (Контекст / Решение / Последствия).
- `<hub>/adr/0001-record-architecture-decisions.md` — пример (мета-ADR: «записывать решения»).
- Хаб `<hub>/adr/` — единый ADR-дом программы.

## Что после

- Ссылка на ADR из `docs/ARCHITECTURE.md` (фаза 10 / правка).
- Если ADR меняет контракт — отдельный PR с правкой `CONVENTIONS` и сервисов
  под него (см. `docs/refs/COMMUNICATION.md`).
