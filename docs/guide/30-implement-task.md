# Этап 30 — Реализовать задачу

**Цель:** выполнить один пункт программного бэклога — от атомарного захвата
глобального лока до merge, auto-rc и освобождения лока.

**Когда:** есть невыполненный пункт в `<hub>/BACKLOG.md`, нужно писать код.

**Что нужно до:** выбран первый доступный пункт `<hub>/BACKLOG.md`; для
service/gateway завершены этапы 10 и 20; для interface/stub заполнен их
`docs/ARCHITECTURE.md`; для хаба определён изменяемый системный контракт или
рабочий артефакт.

**Применимо:** общий рабочий цикл для сервиса, gateway-сервиса, интерфейса,
автономного компонента и хаба. Для interface/stub вместо сервисной спеки
контрактом служит их `docs/ARCHITECTURE.md`; неприменимые роли помечаются `N/A`
по `docs/refs/ROLES.md`. Для задачи хаба целевой контракт — `COMPOSITION.md` /
`CONVENTIONS.md` / `BACKLOG.md` / `adr/`.

**Результат:** PR слит в `main`, задача только после этого отмечена `[x]`,
feature-ветка удалена, создан и развёрнут кандидат `vX.Y.Z-rc.N`.

> **Программный бэклог — строгая последовательная очередь на всю программу**
> (как миграции). Живёт в хабе (`<hub>/BACKLOG.md`), не в сервисе. Запрещено
> выполнять задачи параллельно или перепрыгивать через невыполненный пункт.
> Задача N+1 начинается только после полного завершения задачи N. **Глобальный
> лок:** ровно один пункт `[~]` in-flight на всю программу — пока он есть, новый
> `[ ]` не берётся. Маркер персистентен через коммит в хаб (состояние бэклога =
> состояние гита хаба).

## Шаги

1. **Атомарно взять задачу.** Прочитай актуальный `main` хаба и выбери **самый
   первый** `[ ]` пункт, только если `[~]` отсутствует. Создай короткую ветку
   `feat/backlog-lock-N`, замени `[ ]` на `[~]`, открой PR в хаб и дождись его
   merge. Rule-гейт повторно читает актуальный `main` и принимает lock-PR,
   только если выбранный пункт всё ещё `[ ]`, выше нет невыполненного пункта и
   другого `[~]` нет. Конфликт или изменившийся `main` означает, что лок не
   получен: обнови состояние и не начинай реализацию. Только merge lock-PR
   считается успешным захватом; локальная правка, коммит или открытый PR локом
   не являются. Тег задачи после захвата указывает целевой репозиторий и модуль
   (`<сервис>/<модуль>` → репозиторий + `docs/specs/<module>.md`). Коммит:
   `chore(backlog): mark N in-flight`.
2. **Прочитать контекст.** Для сервиса: `docs/ARCHITECTURE.md` → спека нужного
   модуля (возможно нескольких). Для interface/stub: их `docs/ARCHITECTURE.md`.
   Для хаба: изменяемый системный контракт и связанные ADR.
3. **Ветка от main** в целевом репо:
   ```bash
   git checkout main && git pull && git checkout -b feat/<задача>
   ```
   Модель ветвления — `AGENTS.md` → *Модель ветвления* (коротко: `feat/<задача>`
   от `main`, удаляется после merge; прямой коммит в `main` запрещён; релизы —
   тегами `vX.Y.Z`, `docs/guide/70-release.md`).
4. **Разработчик — код + дым-тесты.** Реализуй по пунктам «Что есть» в спеке.
   Полные adversarial-тесты пишет **тестировщик** (шаг 5), не разработчик.
5. **Тестировщик — adversarial-тесты** (независимый вызов, по спеке, не по
   коду; `docs/refs/ROLES.md`): оспаривают реализацию, не подтверждают.
   Артефакт — в коммит; их проходит `rule` #3.
6. **Проверить перед коммитом** — этап `docs/guide/40-verify.md`
   (lint / test / build + #3 с тестами тестировщика + invariant-чеклист). Все
   блокирующие должны пройти; иначе — bounded фикс-цикл (dev+tester), повтори.
   **Не коммить с упавшим гейтом.**
7. **Обновить контракт:** в сервисной спеке перенеси реализованное из «Что
   TODO» в «Что есть»; для других типов репо актуализируй их рабочий артефакт.
8. **Коммит** — Conventional Commits, scope = имя модуля (или `deploy`/`docs`):
   ```
   feat(<module>): add zfs-snapshot fs probe
   fix(<module>): reject path traversal in HostBridge
   docs: update ARCHITECTURE.md with module matrix
   refactor(<module>): extract envelope signing
   ```
   Breaking changes — `BREAKING CHANGE:` в теле. Язык коммитов — `AGENTS.md` →
   *Язык*.
9. **Пуш, открыть PR в `main`.**
10. **Ревьюер** (если применим; независимый вызов) — conformance #6–10, 13,
    14; adversarial-verify
    → авто-фикс (новый коммит, re-гейт) или `<hub>/BACKLOG`+холд (`docs/refs/ROLES.md`).
11. **Безопасник** (если применим; независимый вызов) — vuln-scan (`rule` #15) + threat-model
    → авто-фикс / `<hub>/BACKLOG`+холд.
12. **Merge по гейту** (без человеческого одобрения) → auto-rc → деплой в
    тестовую среду. Ветка удаляется.
13. **Завершить задачу после merge.** После успешного auto-rc в
    `<hub>/BACKLOG.md` поставь `[~]`→`[x]`, закоммить в feature-ветке хаба и
    влей отдельным PR
    (`chore(backlog): mark N done`). Только `[x]` освобождает глобальный лок и
    разрешает взять следующую задачу. Если работа упёрлась в bounded retries /
    needs-human, верни пункт в `[ ] needs-human` и сдвинь с верхушки вниз — это
    единственное исключение из порядка.

## Источники фактов

- `AGENTS.md` — правила коммитов/ветвления, что можно/нельзя, язык.
- `docs/refs/ROLES.md` — роли агентов (разработчик/тестировщик/ревьюер/безопасник).
- `docs/guide/40-verify.md` — что именно проверять перед коммитом.
- `docs/refs/VERIFICATION.md` — полная теория гейта (если нужно).

## Критерий завершения

Целевой PR слит, auto-rc успешно создан и развёрнут, отдельный PR хаба перевёл
задачу `[~]`→`[x]`, feature-ветки удалены. Только после merge этого последнего
PR глобальный лок освобождён.

## Что после

- Баг из тестовой среды → новый пункт `<hub>/BACKLOG.md`.
- Следующий пункт очереди берётся повторением этапа 30 строго сверху.
