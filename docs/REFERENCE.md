# Краткий справочник

## Статусы задачи

| Статус | Значение |
|---|---|
| `[ ] ready` | первая такая задача доступна агентам |
| `[x]` | merge, test deployment и smoke завершены |
| `needs-input` | требуется продуктовое решение человека |
| `blocked-external` | недоступна внешняя система или доступ |
| `automation-failed` | обязательный инструмент или CI не работает |
| `retry-exhausted` | исчерпан ограниченный цикл исправлений |

Статуса `[~]` и распределённого лока нет.

Канонический заголовок открытой задачи — `### TASK-NNNN. [ ] <status> — Название`,
где `<status>` — `ready` или один из диагностических статусов. Завершённая задача
использует `### TASK-NNNN. [x] — Название`. Диагностика записывается сразу под
заголовком; поля цели, критериев и scope сохраняются.

## Риск и автономность

| Риск | Примеры | Максимальное действие агента |
|---|---|---|
| `low` | документация, локальный рефакторинг | merge и test deployment |
| `medium` | обратимо совместимая функция без изменения данных | merge и test deployment |
| `high` | auth, права, PII, публичный контракт, миграция | test deployment; человек разрешает production |
| `critical` | удаление данных, платежи, production-секреты, необратимое действие | человек до merge и перед deploy |

Агент может повысить риск и понизить автономность, но не наоборот. Он не может
ослабить гейт, принять остаточный риск или расширить собственные полномочия.
Неопределённость трактуется в сторону более строгого класса. Недоступная
обязательная проверка блокирует работу; fail-open допустим только для явно
необязательной проверки `low`-задачи.

## Триггеры проверок

| Изменение | Дополнительная проверка |
|---|---|
| Документация, локальный рефакторинг | rule gate |
| Обычная функциональность | независимое review |
| Сложные состояния или интеграция | сценарный/test lens |
| API, event schema | compatibility + contract tests |
| Auth, permissions, secrets, сеть, файлы, ввод | security lens + scan |
| Persisted data | migration dry run + rollback/roll-forward |
| Заданный человеком новый репозиторий или системная граница | architecture review; возможно ADR |

Триггер повышает минимальный риск: security, публичный контракт и persisted data
не могут быть ниже `high`; необратимое действие не может быть ниже `critical`.

## Команды стеков

| Стек | Проверка | Тест | Сборка |
|---|---|---|---|
| Python | `ruff format --check . && ruff check . && pyright` | `pytest` | `uv build` |
| Go | `gofmt -l . && go vet ./...` | `go test ./...` | `go build ./...` |
| Rust | `cargo fmt --check && cargo clippy -- -D warnings` | `cargo test` | `cargo build --release` |
| TypeScript | `pnpm lint && tsc --noEmit` | `pnpm test` | `pnpm build` |

Репозиторий выбирает один стек и фиксирует точные команды в своём `AGENTS.md`.

## Git

- Ветка: `feat/TASK-NNNN-<slug>`.
- Коммиты: Conventional Commits.
- Интеграция: PR → обязательные checks → squash merge.
- По умолчанию агент сам выполняет push feature-ветки, создаёт PR, сливает
  зелёный PR в удалённый `main` и синхронизирует локальный `main`; полный порядок
  и условия остановки заданы в `WORKFLOW.md`.
- Релиз: annotated tag `vX.Y.Z`; RC только по необходимости.
- Прямые коммиты в `main`, `dev` и release-ветки запрещены.

## Задача

```markdown
### TASK-NNNN. [ ] ready — Краткое название

Цель:
Наблюдаемый результат для пользователя или системы.

Готово, когда:
- проверяемый критерий;
- проверяемый критерий.

Не входит:
- явное ограничение или «нет».
```

Target, технический план, тесты, risk triggers и rollback квалифицирует агент.
