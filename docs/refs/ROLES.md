# Роли агентов (референс)

Автономный цикл исполняют **четыре роли** — разработчик, тестировщик,
ревьюер, безопасник. Каждая роль — отдельный вызов агента со **свежим
контекстом**; роли работают **строго последовательно** (каждая блокирует
следующую), без параллелизма внутри задачи. Роль оправдана, только если
владеет классом ошибок, который никто другой не ловит, и имеет метод
(детерминированный артефакт → `rule`, либо отдельный adversarial-lens →
conformance).

> Референс (факт «как устроен агент-слой»). Применим к инстанцированным репо
> (хаб/сервисы/интерфейсы/stub); сам репо методологии — исключение (его PR'ы
> ревьюит человек). Цикл — `docs/refs/PIPELINE.md`; гейт —
> `docs/refs/VERIFICATION.md`; процедура перед коммитом —
> `docs/guide/40-verify.md`.

## Принцип независимости

Ревьюер, тестировщик и безопасник — **не тот же вызов, что разработчик**, и
читают **спеку/контракт, а не код разработчика** (там, где это уместно).
Само-ревью / само-тестирование одним вызовом подтверждает предположения
автора, а не оспаривает их — ценность низкая. Независимый вызов + другой
источник = единственный способ поймать ошибку исполнителя без человека.

## Роли

| Роль | Что делает | Артефакт / lens | Куда в гейте | Класс ошибок |
|---|---|---|---|---|
| **разработчик** | код + дым-тесты по спеке | код | — | реализация |
| **тестировщик** | adversarial-тесты **по спеке, не по коду** | тесты → `rule` #3 | блокирует (CI гоняет) | same-author tests (подтверждают, не оспаривают) |
| **ревьюер** | conformance: код↔спека, швы, направление зав-тей, ARCHITECTURE↔хаб | finding → conformance | adversarial-verify → авто-фикс/`BACKLOG` | структурное соответствие |
| **безопасник** | vuln-scan + threat-model | vuln-scan → `rule` #15; threat-model → conformance | блокирует (scan) + авто-фикс/`BACKLOG` | security-класс (compiles+passes, но дыряво) |

Ревьюер — это и есть существующий conformance-проход (#6–10, 13, 14);
отдельного «ревьюера поверх ревьюера» не заводить. 5-я роль требует
обоснования по принципу выше — без уникального класса ошибок не заводится.

## Цепочка (строго последовательно, каждая блокирует следующую)

```
разработчик → код + дым-тесты (по спеке)
тестировщик → adversarial-тесты (по спеке, не по коду) → в коммит
rule-гейт #1–5, 11, 12, 15-scan + #3(тесты тестировщика) — блокирует;
   dev+tester bounded фикс-цикл, re-гейт
ревьюер    → conformance #6–10, 13, 14; adversarial-verify → авто-фикс / BACKLOG
безопасник → vuln-scan(rule) + threat-model; adversarial-verify → авто-фикс / BACKLOG
всё зелёное → auto-merge → auto-rc → test-deploy
```

Каждый шаг — предшественник следующего; роль N+1 не стартует, пока N не
завершён и (для rule/conformance) не пройден. Bounded retries: фикс-цикл
ограничен N попытками; дальше — `BACKLOG`, цепочка не зацикливается.

> **Между задачами — тот же принцип на уровне программы.** Ровно одна задача
> in-flight на всю программу (глобальный лок в `<hub>/BACKLOG.md`, маркер `[~]`,
> персистентен через коммит в хаб). Пока `[~]` есть, новый `[ ]` не берётся —
> агенты работают строго друг за другом, без гонок. Модель —
> `docs/refs/PIPELINE.md`.

Порядок выбран так: тестировщик **до** коммита (его тесты — в коммите, `rule`
#3 гоняет их на гейте); ревьюер и безопасник — **после** пуша (на PR): они
судят дифф, их находки — авто-фикс новым коммитом или `BACKLOG`+холд.

## Детерминированный пол прежде всего

Для каждой роли вопрос: можно ли это сделать `rule`? Если да — `rule`
(блокирует железно), не LLM. Тестировщик: его тесты — `rule` #3 (CI гоняет).
Безопасник: vuln-scan — `rule` #15. LLM-часть — только то, что сканер/линтер
не ловит (логика, threat-model, соответствие кода спеке). Чем шире пол, тем
меньше решений на недетерминированном слое.

## Применимость по типу репо

- **Сервис:** все 4 роли; gateway — безопасник обязателен (доверительная
  граница, presentation-эндпоинты).
- **Интерфейс:** разработчик + тестировщик + ревьюер; безопасник — частично
  (XSS, секреты в браузере, `VITE_`-токены). `MODULE`/`SPEC` не применяются,
  но тестировщик пишет компонентные/e2e тесты по `ARCHITECTURE`-манифесту.
- **Stub-таргет:** разработчик + безопасник (по поверхностям, если есть);
  тестировщик/ревьюер — по форме (есть ли наблюдаемая поверхность).

## Что роли НЕ закрывают

- **Неточные спеки.** Тестировщик пишет **правильные тесты на неправильную
  спеку** — он читает спеку, не угадывает волю. Правит спеку только человек
  (его домен). Роли поднимают пол, не заменяют точный бэклог.
- **Классы багов, которые не ловит ни сканер, ни тест-среда** (тонкие ошибки
  абстракции, некоторые edge-case'ы под нагрузкой). Их ловит только баг в
  тесте/проде → `BACKLOG` (модель принимает этот сдвиг).
