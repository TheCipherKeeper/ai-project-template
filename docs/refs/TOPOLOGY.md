# Топология репозиториев (референс)

Программа — это **хаб + N сервисов**, по репозиторию на каждый. Этот репо —
**методология** (центральный авторитет): он читается, а инстанции
(хаб/сервисы) создаются копированием `skeletons/`.

> Референс (факт «как устроена топология»). Процедуры — в `docs/guide/`,
> модель верификации рёбер — в `docs/refs/VERIFICATION.md`.

## Слои

| Слой | Репо | Несёт | Роль |
|---|---|---|---|
| **Методология** (этот репо) | один | `docs/guide/`, `docs/refs/`, `skeletons/`, корневые `README`/`AGENTS` | читается; скелеты копируются; корневой авторитет для гейта |
| **Хаб** | один | `COMPOSITION.md`, `CONVENTIONS.md`, системный `docker-compose.yml`, `adr/` | состав программы, кросс-сервисные контракты (event envelope), системный compose, ADR-дом |
| **Сервис** | по репо на сервис | `AGENTS.md`, `docs/{ARCHITECTURE,BACKLOG,specs/,adr/}`, `Dockerfile`, `docker-compose.yml` (локальный) | один микросервис; клиент брокера; инстанциация из `skeletons/service/` |

## Что где живёт

- **Кросс-сервисные контракты** (event envelope, состав программы, системный
  compose, ADR) — в **хабе**, не в сервисах.
- **Архитектура/бэклог/спеки одного сервиса** — в **сервис-репо**
  (`docs/ARCHITECTURE.md`, `docs/BACKLOG.md`, `docs/specs/<module>.md`).
- **Процедуры и факты методологии** — в этом репо (`docs/guide/`, `docs/refs/`),
  читаются сервисами/хабом централизованно, **не копируются** в них.
- **Стартовые файлы** (заготовки) — в `skeletons/service/` и `skeletons/hub/`.

## Правила

- Один репо = один сервис (или хаб). Не смешивать сервисы в одном репо.
- Сервис-репо **не несёт** `docs/guide/`/`docs/refs/` — ссылается на этот репо
  методологии (см. `skeletons/service/AGENTS.md`).
- Прямая service-to-service связность в обход брокера — запрещена; это выход
  за границу сервиса и решается в хабе/ADR (см. `docs/refs/COMMUNICATION.md`).
- ADR: для проекта с хабом — в хабе (`<hub>/adr/`); ADR-ссылки из сервисов
  указывают туда. Standalone-сервис без хаба — `docs/adr/` в своём репо
  (скелет — `skeletons/service/docs/adr/`).

## Edge-модель (верификация)

Гейт проверяет рёбра `методология → хаб → сервисы`: при изменении в узле — все
инцидентные рёбра (вверх — соответствие авторитету, вниз — все дочерние
соответствуют ему на пиннённой версии `CONVENTIONS@v<N>`). Список детей «вниз» —
это `COMPOSITION.md` хаба. Полная схема — `docs/refs/VERIFICATION.md`.

## Инстанциация

- **Новый сервис:** скопируй `skeletons/service/` → новый репо → выбери стек
  (`docs/guide/00-bootstrap.md`) → заполни `ARCHITECTURE`/`BACKLOG`/`specs`.
- **Новый хаб:** скопируй `skeletons/hub/` → новый репо → заполни
  `COMPOSITION`/`CONVENTIONS` → добавь сервисы по мере их появления.
