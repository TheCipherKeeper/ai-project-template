# Канон структуры спеки (референс)

Спек описывает **контракт модуля** (что), не реализацию (как). Один модуль
workspace'а = один файл `docs/specs/<module>.md` по этой структуре.
Заполняемый пример — `docs/specs/EXAMPLE.md` (копируй под свой модуль).

> Это референс (как должна быть устроена спека). Процедура заведения модуля
> и написания спеки — пошагово в `docs/guide/20-define-module.md`.

## Секции (порядок фиксирован)

1. **Описание** — кратко: какую роль играет модуль, границы ответственности.
2. **Интерфейсы** — публичные функции/методы модуля: имена, сигнатуры, что делают.
3. **Типы** — публичные типы модуля в синтаксисе выбранного стека
   (Rust `enum/struct`, Python `dataclass`/`Protocol`, Go `type`/`interface`,
   TypeScript `interface`/`type`). Ошибки — здесь же.
4. **Что есть** — реализованное поведение. **Каждый пункт → тест.**
5. **Что TODO** — не реализовано. Эти пункты переезжают в «Что есть» по мере
   реализации, а в `docs/BACKLOG.md` ставится `[x]`.
6. **Ограничения** — чего модуль не делает; явные запреты (важно для агентов,
   чтобы не «помогали» там, где не надо).
7. **Зависимости** — внутренние модули и внешние библиотеки выбранного стека.

## Чек-лист

- Прочитай `docs/ARCHITECTURE.md` перед работой с любым модулем.
- Прочитай спеку модуля перед изменением кода в нём.
- После реализации — обнови спеку: перенеси пункт из «Что TODO» в «Что есть».
- Каждый пункт «Что есть» → должен иметь тест.
- Спека описывает контракт (что), не реализацию (как).
- Как соотнести модуль с каталогами выбранного стека — `docs/refs/LAYOUT.md`.
