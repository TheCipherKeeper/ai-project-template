# Канон структуры спеки (референс)

Спек описывает **контракт модуля** (что), не реализацию (как). Один модуль
workspace'а = один файл `docs/specs/<module>.md` по этой структуре.
Заполняемый пример — `docs/specs/EXAMPLE.md` (копируй под свой модуль).

> Референс (как должна быть устроена спека). Процедура заведения модуля —
> пошагово в `docs/guide/20-define-module.md`. Внутренняя архитектура модуля
> (usecases/ports/domain/adapters) — `docs/refs/MODULE.md`; спека описывает
> «что» делает каждый юзкейс, не «как» (handler).

## Секции (порядок фиксирован)

1. **Описание** — кратко: какую роль играет модуль, границы ответственности.
2. **Интерфейсы** — **юзкейсы** модуля. На каждый юзкейс: input port
   (сигнатура `execute(In) -> Out`/ошибки) и потребляемые им output ports
   (из `ports/`, см. `docs/refs/MODULE.md`).
3. **Типы** — доменные типы и per-usecase DTO в синтаксисе выбранного стека
   (Rust `enum/struct`, Python `dataclass`/`Protocol`, Go `type`/`interface`,
   TypeScript `interface`/`type`). Ошибки — здесь же.
4. **Что есть** — реализованное поведение, **по юзкейсам**. Каждый пункт → тест.
5. **Что TODO** — не реализовано, по юзкейсам. Эти пункты переезжают в «Что
   есть» по мере реализации, а в `docs/BACKLOG.md` ставится `[x]`.
6. **Ограничения** — чего модуль не делает; явные запреты (важно для агентов,
   чтобы не «помогали» там, где не надо).
7. **Зависимости** — output ports + внутренние модули + внешние библиотеки
   выбранного стека.

## Чек-лист

- Прочитай `docs/ARCHITECTURE.md` перед работой с любым модулем.
- Прочитай спеку модуля перед изменением кода в нём.
- После реализации — обнови спеку: перенеси пункт из «Что TODO» в «Что есть».
- Каждый пункт «Что есть» → должен иметь тест.
- Спека описывает контракт (что) каждого юзкейса, не реализацию (как).
- Швы и направление зависимостей модуля — `docs/refs/MODULE.md`.
- Как соотнести модуль с каталогами выбранного стека — `docs/refs/LAYOUT.md`.
