# Контроль соответствия — справочник

Автоматическая проверка соответствия репозитория каноническим требованиям.
Выполняется для каждого коммита.

> Документ определяет проверяемые связи, виды проверок, перечень инвариантов и
> правила запуска. Пошаговая проверка перед коммитом описана в
> `docs/guide/40-verify.md`.

При изменении репозитория проверяются все связанные с ним отношения:

- **вверх** — соответствие репозитория вышестоящему источнику правил;
- **вниз** — соответствие зависимых репозиториев изменённому источнику правил.

Такой порядок обеспечивает фактическое применение правил и выявляет
расхождения между источником и зависимыми репозиториями. Правила читаются из
канонического источника, а не из проверяемой копии.

> Этот репозиторий содержит **методологию** и является корневым источником
> правил. Для него проверяются только связи
> только ребро **вниз** (`методология → хаб` + `методология → все интерфейсы` +
> `методология → все автономные компоненты`; сервисы — транзитно через хаб): при изменении
> канона все инстанции перепроверяются. Ребро «вверх» у корня
> с вышестоящим источником нет. Сам репозиторий проходит **проверку документов**
> (проверка доков/скелетов, см. корневой `AGENTS.md`). Правило обязано быть
> единым по всему графу, иначе гейт не композируется.

## Рёбра и направления

| Где коммит | Проверяемые рёбра | Что проверяется |
|---|---|---|
| Методология (этот репо) | вниз: `методология → хаб` + `методология → все интерфейсы` + `методология → все автономные компоненты` | все экземпляры соответствуют обновлённому канону (сервисы — через хаб) |
| Хаб | вверх: `методология → хаб`; вниз: `хаб → все сервисы` + `хаб → все интерфейсы` + `хаб → все автономные компоненты` | хаб соответствует методологии; все сервисы соответствуют хабу; интерфейсы и автономные компоненты соответствуют `COMPOSITION` |
| Сервис | вверх: `хаб → сервис`; presentation (только для gateway): `gateway → интерфейсы` | сервис соответствует хабу (`CONVENTIONS`); для gateway — его presentation-эндпоинты соответствуют тому, что заявляют интерфейсы (agent) |
| Интерфейс | вверх: `методология → интерфейс` + `хаб → интерфейс`; presentation: `gateway → интерфейс` | интерфейс соответствует методологии (структура/канон) и `COMPOSITION` (потребляет только существующие эндпоинты gateway); его заявленные вызовы соответствуют `ARCHITECTURE` gateway (agent) |
| Автономный компонент | вверх: `методология → stub` + `хаб → stub` | компонент соответствует методологии и зарегистрирован в `COMPOSITION`; `MODULE` и `SPEC` не применяются; компонент не подключается к брокеру |

Список детей для «вниз» — это `COMPOSITION.md` хаба (отдельный реестр не нужен).

## Источник правил — авторитет, не копия

Гейт читает правила из **авторитета** (методология/хаб), а не из копии в
проверяемом репо. Иначе копия, уплывшая от канона, проверяет сама себя и
расхождение не видит. Поэтому сервисы/хаб **не несут** `docs/guide/`/`docs/refs/`
и ссылаются на этот репо методологии, а не копируют их. Это ключевое условие
закрытия проблемы рассинхронизации.

## Виды проверок

| Тип | Что проверяется | Кто | Блокирует? | Разрешение находки |
|---|---|---|---|---|
| **Проверка работоспособности** | сборка, тесты и соблюдение исполняемых контрактов | воспроизводимые проверки CI | да | исправить нарушение до коммита |
| **Проверка соответствия** | структура, документация, контракты, темы брокера и ссылки на ADR | программный агент | сырой вердикт не блокирует | независимо перепроверить вывод; затем исправить нарушение либо внести задачу в `BACKLOG` и приостановить запрос на слияние |

Проверку работоспособности выполняет CI: для неё требуется воспроизводимый
результат, а не оценка языковой модели. Сырой вердикт агента о соответствии не
блокирует изменения, поскольку возможно ложное срабатывание. Подтверждённая
находка обязана быть разрешена и может удерживать PR.
Находку должен подтвердить независимый проверочный запуск с задачей опровергнуть
первоначальный вывод. Подтверждённое нарушение исправляют новым коммитом либо
регистрируют в `BACKLOG` с приостановкой запроса на слияние. Число повторных
попыток исправления ограничено; после исчерпания попыток создаётся задача в
`BACKLOG`.

> **Кто ревьюит.** В автономном цикле ревьюер — независимый вызов агента (не
> имплементер); человек участвует только в risk-gate, когда это требует класс
> риска. Сам репо методологии —
> исключение: его гейт docs-verify, его PR'ы ревьюит человек (`AGENTS.md`).

## Перечень инвариантов

Перечень инвариантов обязателен. Общего указания «проверить соответствие»
недостаточно: оно не задаёт воспроизводимый объём проверки.

| ID | Инвариант | Тип | Блок. | Скоуп |
|---|---|---|---|---|
| VER-001 | Локальная работа идёт в `feat/<task>`; `main` проверяется как CI-контекст после merge | rule | да | дифф |
| VER-002 | Conventional Commits + scope | rule | да | дифф |
| VER-003 | lint/test/build проходят; `AC-N` трассируются в выполненные tests/probes | rule (behavioral) | да | репо |
| VER-004 | Дифф тронул спеку «TODO→есть» → тронул код/тест модуля | rule | да | дифф |
| VER-005 | Только первый `[ ] ready` после DoR атомарно меняется на `[~]`; не более одного `[~]`; `[x]` только после post-deploy evidence; диагностический punt — единственное исключение | rule | да | вся система |
| VER-006 | Каждый пункт «Что есть» и каждый acceptance criterion имеют тест/probe | agent | нет | дифф+ |
| VER-007 | Содержимое «Что есть» соответствует коду | agent | нет | cross-cutting |
| VER-008 | ARCHITECTURE (роль/топики/граница) соответствует хабу (`CONVENTIONS`) | agent | нет | cross-cutting |
| VER-009 | Заглушки и незавершённые части явно обозначены и не выдаются за готовую реализацию | agent | нет | несколько репозиториев |
| VER-010 | Доки не противоречат внутри яруса; между ярусами старший побеждает | agent | нет | cross-cutting |
| VER-011 | ADR-ссылки существуют; JSON/YAML-схемы валидны | rule | да | cross-cutting |
| VER-012 | Модуль экспонирует каноничные швы; каждый юзкейс определяет input port | rule (structural) | да | модуль |
| VER-013 | Направление зависимостей соответствует `MODULE.md` | agent | нет | модуль+ |
| VER-014 | Интерфейс зарегистрирован в хабе; consumed-эндпоинты существуют у gateway | agent | нет | cross-cutting |
| VER-015 | Dependency/secret scan проходит; threat-model не содержит неразрешённых дыр | rule + agent | да | cross-cutting |
| VER-016 | Risk/autonomy не занижены; обязательные роли завершены fail-closed | rule + agent | да | задача |
| VER-017 | Wire-контракты валидны и совместимы; contract tests проходят | rule | да | cross-cutting |
| VER-018 | Миграция имеет совместимый план, dry run и rollback/roll-forward | rule + agent | да | данные |
| VER-019 | Post-deploy readiness/smoke/probes и окно наблюдения зелёные | rule | да | среда |
| VER-020 | Evidence полон, связан с task/run/commit и не содержит секретов | rule | да | задача |

`rule` — детерминированная проверка (скрипт/CI). `agent` — семантическая (LLM).
`cross-cutting` — требует чтения за пределами диффа. **Роли агентов**
(разработчик/тестировщик/ревьюер/безопасник), их lens, артефакты и место в
гейте — `docs/refs/ROLES.md`; тестировщик кормит `VER-003`, безопасник —
`VER-015`. Канон швов модуля — `docs/refs/MODULE.md`; процедура заведения —
`docs/guide/20-define-module.md`. Канон интерфейса как клиента —
`docs/refs/COMMUNICATION.md` → *Пользовательский интерфейс*; шаблон — `skeletons/interface/`.
Канон автономного компонента — `docs/refs/COMMUNICATION.md` →
*Автономный компонент*; шаблон — `skeletons/stub/`.

## Применимость инвариантов по типу репо

Перечень ориентирован на сервисы, но на него ссылаются также интерфейсы и автономные компоненты —
поэтому явно:

- **Сервис:** весь чеклист (#1–15). `gateway-сервис` — тоже сервис (весь
  чеклист), но **цель** #14 (presentation-эндпоинты, потребляемые интерфейсами)
  применима **только** к нему (см. `COMMUNICATION` → *Сервис-шлюз*), а
  **#15 (безопасник)** для gateway обязателен (доверительная граница).
  У обычных сервисов #14-цель N/A; их клиентский край — топики брокера,
  потребляемые gateway (проверяет ребро `хаб → сервис`, инвариант #8).
- **Интерфейс:** #1, 2, 3, 9, 10, 11 (ветвление/коммиты, lint/test/build, stub,
  противоречие доков, ADR-ссылки) + #14 (регистрация в `COMPOSITION` и
  соответствие заявленных consumed-эндпоинтов `ARCHITECTURE` **gateway-сервиса**)
  + #15-частично (безопасник: XSS, секреты в браузере, `VITE_`-токены).
  `MODULE.md`/`SPEC.md` (#4–8, 12, 13) — бэкенд-канон, к интерфейсу не
  применяются.
- **Автономный компонент:** #1, 2, 3, 9, 10, 11 (общие требования к ветвлению и коммитам,
  lint/test/build, stub, противоречие доков, ADR-ссылки) + #15 (безопасник по
  поверхностям, если есть). Без сервисных (#4–8, 12, 13 —
  `MODULE`/`SPEC`/specs/швы не применяются: stub параметризуется
  дескриптором, не юзкейсами) и без интерфейсных (#14 — у stub нет
  consumed-эндпоинтов). Stub не потребляет envelope, хаб-контракт к нему
  неприменим. Регистрация stub'а в `COMPOSITION` — как #14 для интерфейса, agent.
  Бэклог — программный, в хабе (`<hub>/BACKLOG.md`); своего `BACKLOG` stub не
  несёт (как и любой репо, кроме хаба).

## Начальная проверка

До включения автоматического контроля выполняется однократная полная проверка
`хаб → все сервисы`. Она фиксирует исходное состояние и список известных
расхождений, но не блокирует работу. Это предотвращает появление множества ранее
существовавших ошибок при первом изменении хаба.

## Триггер и политики

- Триггер — **на каждый коммит**, включая merge-commit и прямой push в
  `main`, иначе дрейф вносится в обход гейта.
- Детерминированные проверки (#1, 2, 3, 4, 5, 11, 12, 15-scan) — блокируют.
- Агентские conformance-проверки (#6, 7, 8, 9, 10, 13, 14, 15-threat-model) —
  сырой вердикт не блокирует; подтверждённая находка разрешается (авто-фикс /
  `BACKLOG`+холд). Ошибка/недоступность агента **не** блокирует. Цикл
  фикс→recheck ограничен числовой политикой `docs/refs/RETRY.md`; дальше —
  диагностический статус, evidence и hold.
- Каденс: rule — на push (дёшево); agent — на PR (готов к ревью), не на каждый
  коммит.
- **Auto-rc (автономный цикл):** merge в `main` → авто-тег `vX.Y.Z-rc.N` →
  деплой в тестовую среду. Тестовая среда — первая точка человека (баг →
  `BACKLOG`). Релизная процедура — `docs/guide/70-release.md`; модель цикла —
  `docs/refs/PIPELINE.md`.
