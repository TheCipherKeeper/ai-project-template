# Верификация (verification gate) — референс

Автоматическая проверка соответствия репозитория канону на каждый коммит.

> Это референс (полная теория: рёбра графа, версионирование контрактов,
> conformance/behavioral, полный invariant-чеклист, триггер/политики).
> Процедура «проверить перед коммитом» — компактно в `docs/guide/40-verify.md`.

**Идея:** на изменение в узле графа проверять все рёбра, инцидентные этому
узлу — **вверх** (узел соответствует авторитету) и **вниз** (все дочерние узлы
соответствуют ему). Это закрывает две проблемы разом: enforcement (правила
проверяются, а не существуют на бумаге) и рассинхронизацию копии с
первоисточником (правила читаются из авторитета, а не из копии, и каждое
изменение — это коммит, который триггерит нужное ребро).

> Этот репо — **листовой сервис** в графе (один репо = один сервис, под хабом
> `cybercity`). Для него активно только ребро **вверх** (сервис ↔ хаб). Правило
> «вниз» зафиксировано для узлов-родителей (хаб, шаблон) — здесь оно неактивно,
> но обязано быть единым по всему графу, иначе гейт не композируется.

## Рёбра и направления

| Где коммит | Проверяемые рёбра | Что проверяется |
|---|---|---|
| Этот сервис | вверх: `хаб → сервис` | сервис соответствует хабу на пиннённой версии контрактов |
| Хаб `cybercity` | вверх: `шаблон → хаб`; вниз: `хаб → все сервисы` | хаб соответствует шаблону; все сервисы соответствуют хабу |
| Шаблон `ai-project-template` | вниз: `шаблон → хаб` + `шаблон → все сервисы` | все инстанции соответствуют обновлённому канону |

Список детей для «вниз» — это `COMPOSITION.md` хаба (отдельный реестр не нужен).

## Источник правил — авторитет, не копия

Гейт читает правила из **авторитета** (хаб/шаблон), а не из копии в
проверяемом репо. Иначе копия, уплывшая от канона, проверяет сама себя и
расхождение не видит. Это ключевое условие закрытия проблемы рассинхронизации.

## Версионирование контрактов (обязательно для «вниз»)

Без версионирования ребро «вниз» нерабочее: изменение хаба красит все сервисы
одновременно, а закоммитить скоординированный апдейт N репо атомарно нельзя.

- Хаб экспонирует контракты с версиями: `CONVENTIONS@v1`, `@v2`, …
- Сервис **пинит** версию в своей конфигурации/манифесте
  (см. `docs/ARCHITECTURE.md` → *Контракты хаба*).
- Гейт проверяет сервис против **пиннённой** версии, а не HEAD.
- Изменение хаба в `@v2` не красит сервисы на `@v1`.
- Миграция — отдельным PR сервиса: бамп пина `v1→v2` + правки под новый контракт.

## Conformance vs behavioral

| Тип | Что проверяется | Кто | Блокирует? |
|---|---|---|---|
| **conformance** | структурное соответствие: доки, методология, присутствие контракта, топики, ADR-ссылки | LLM-агент | нет (консультативно, PR-комментарии) |
| **behavioral** | собирается/тестируется ли сервис против контрактов | детерминированный CI (lint/test/build) | да |

Гейт-агент не должен делать behavioral — там нужна воспроизводимость билда,
не суждение LLM. Агент-ошибка не блокирует; путь апелляции (false positive)
обязателен — иначе доверие к гейту сдувается.

## Invariant-чеклист

Без явного чеклиста «агент проверяет соответствие сверху вниз» — размытое
требование, которое агент трактует по-разному каждый раз (это и убивает
детерминизм). Чеклист обязателен.

| # | Инвариант | Тип | Блок. | Скоуп |
|---|---|---|---|---|
| 1 | Ветка `feat/<task>` (или main) | rule | да | дифф |
| 2 | Conventional Commits + scope | rule | да | дифф |
| 3 | lint/test/build проходят | rule (behavioral) | да | репо |
| 4 | Дифф тронул спеку «TODO→есть» → тронул код/тест модуля | rule | да | дифф |
| 5 | `[x]` в BACKLOG — на первом невыполненном пункте (без перепрыгивания) | rule | да | cross-cutting |
| 6 | Каждый пункт «Что есть» в спеке → есть тест | agent | нет | дифф+ |
| 7 | Содержимое «Что есть» соответствует коду | agent | нет | cross-cutting |
| 8 | ARCHITECTURE (роль/топики/граница) соответствует хабу@pin | agent | нет | cross-cutting |
| 9 | Сервис пинит версию контракта; версия ещё поддерживается хабом | rule | да | cross-cutting |
| 10 | Stub не выдаётся за реализацию (placeholder/TODO помечены) | agent | нет | cross-cutting |
| 11 | Доки не противоречат; старший приоритет побеждает | agent | нет | cross-cutting |
| 12 | ADR-ссылки из ARCHITECTURE существуют в хабе | rule | да | cross-cutting |

`rule` — детерминированная проверка (скрипт/CI). `agent` — семантическая (LLM).
`cross-cutting` — требует чтения за пределами диффа.

## Базовый аудит

Перед включением гейта репозитории уже могли дрейфовать. Нужен одноразовый
полный проход `хаб → все сервисы` (неблокирующий), фиксирующий baseline и
список расхождений. Иначе первый же коммит хаба зажжёт всё красным.

## Триггер и политики

- Триггер — **на каждый коммит**, включая merge-commit и прямой push в
  `main`, иначе дрейф вносится в обход гейта.
- Детерминированные проверки (#1, 2, 3, 4, 5, 9, 12) — блокируют.
- Агентские conformance-проверки (#6, 7, 8, 10, 11) — консультативны;
  ошибка/недоступность агента **не** блокирует.
- Каденс: rule — на push (дёшево); agent — на PR (готов к ревью), не на каждый
  коммит.
