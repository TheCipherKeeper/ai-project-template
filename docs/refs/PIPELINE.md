# Автономный цикл разработки — справочник

Программа работает **без человека в pre-deploy-цикле**. Ревьюер и верификатор —
агенты; человек входит в систему ровно двумя способами: **(1)** пишет
программный бэклог в хабе (`<hub>/BACKLOG.md` — что делать по всей программе),
**(2)** разбирает баг, всплывший в тестовой/прод-среде, и заводит его задачей в
`<hub>/BACKLOG.md` (далее — обычный цикл). Всё между — агенты. Тестовая среда —
первая точка, где человек видит результат.

> Референс (факт «как устроен цикл без человека»). Применим к
> созданным репозиториям хаба, сервисов, интерфейсов и автономных компонентов. Сам репозиторий
> методологии сюда **не** относится — его гейт docs-verify, его PR'ы ревьюит
> человек (`AGENTS.md`). Процедура работы над задачей —
> `docs/guide/30-implement-task.md`; гейт — `docs/guide/40-verify.md` и
> `docs/refs/VERIFICATION.md`; релизы — `docs/guide/70-release.md`.

## Последовательность

```
<hub>/BACKLOG (человек пишет; программа-wide, глобальный лок) ─▶ [разработчик]
   ветка feat в целевом репо → код + дым-тесты (по спеке)
   ▼
[тестировщик — ДРУГОЙ вызов, по спеке, не по коду] adversarial-тесты → в коммит
   ▼
[гейт] rule (#1–5, 11, 12, 15-scan) + #3(тесты тестировщика) — блокирует
   железно; fail → dev+tester bounded фикс-цикл, re-гейт
   ▼  rule зелёный
[ревьюер — ДРУГОЙ вызов] conformance (#6–10, 13, 14) → adversarial-verify →
   · не выжила       → выкинуть (false positive)
   · выжила, структурная   → авто-фикс (новый коммит, re-гейт)
   · выжила, неструктурная → в <hub>/BACKLOG, PR на холд
   ▼
[безопасник — ДРУГОЙ вызов] threat-model (#15) → adversarial-verify →
   авто-фикс / <hub>/BACKLOG+холд
   ▼  чисто
auto-merge в main → auto-rc (vX.Y.Z-rc.N) → деплой в тестовую среду
   ▼  [человек — впервые]
баг в тесте/проде → задача в <hub>/BACKLOG → обратно в петлю как пункт
```

Глобальный лок: пункт идёт `[ ]`→`[~]`→`[x]` в `<hub>/BACKLOG.md`; `[~]`
коммитится в хаб (`chore(backlog): …`), так что состояние бэклога = состояние
гита хаба, видимое всем репо/агентам. Роли (lens, артефакт, место в гейте,
применимость) — `docs/refs/ROLES.md`.

## Участие человека

1. **Пишет `<hub>/BACKLOG.md`** — единственный источник «что делать» по всей
   программе. Строгая очередь сверху вниз (`docs/guide/30-implement-task.md`).
2. **Разбирает баги из тест/прода** — и заводит их задачами в `<hub>/BACKLOG.md`.
   Дальше баг идёт по той же петле, что и любая задача.

Человек не пишет код, не ревьюит PR, не мержит, не ставит теги — всё это
делают агенты по гейту.

## Требования к программным агентам

- **Четыре роли, строго последовательно.** Разработчик → тестировщик →
  ревьюер → безопасник, каждый — независимый вызов, каждый блокирует
  следующий. Роли, их lens, артефакты и место в гейте — `docs/refs/ROLES.md`.
- **Глобальный лок между задачами.** Ровно один пункт `[~]` in-flight на всю
  программу (в `<hub>/BACKLOG.md`). Пока `[~]` есть, новый `[ ]` не берётся —
  агенты работают строго друг за другом, медленно, но без гонок. Маркер
  персистентен через коммит в хаб, не в памяти агента. Punt (needs-human) —
  задача съезжает с верхушки, единственное исключение из «без перепрыгивания».
- **Независимый вызов, не само-ревью.** Ревьюер/тестировщик/безопасник — не
  тот же вызов, что разработчик; читают спеку/контракт, а не его код. Само-
  ревью одним вызовом подтверждает предположения автора, не оспаривает —
  ценность низкая.
- **Adversarial-verify перед действием.** Любая conformance-находка
  подтверждается попыткой её опровергнуть, прежде чем действовать (авто-фикс
  или в `BACKLOG`). Без человека фильтр false positive должен стоять **до**
  действия, не после.
- **Разрешение находки — два исхода, не «комментарий».** Авто-фикс (новый
  коммит → re-гейт) либо запись в `BACKLOG` + холд PR. «Комментарий в воздух»
  не допускается — в автономном цикле его некому читать.
- **Bounded retries.** Цикл «фикс → recheck → новая находка» ограничен N
  попытками; дальше — STOP, запись в `BACKLOG`, холд. Иначе два LLM
  раскручиваются на ложной находке бесконечно.
- **Детерминированный пол — максимально широкий.** Всё, что можно проверить
  скриптом, — `rule` (блокирует железно), не `agent`. Чем шире пол, тем меньше
  решений ложится на недетерминированный агент-слой. Кандидаты на промоушен —
  `docs/refs/VERIFICATION.md` → *Conformance vs behavioral*.

## Неизменяемые правила

- `rule` (#1–5, 11, 12) блокирует железно. Человек не нужен, чтобы пройти
  детерминированный гейт.
- Прямой коммит в `main` запрещён; путь — feature-ветка + PR + merge. Merge
  делает агент по зелёному гейту, не человек.
- Релизы — тегами `vX.Y.Z-rc.N` (pre-release) и `vX.Y.Z` (стабильная);
  release-веток нет (`docs/guide/70-release.md`). В автономном цикле rc режется
  автоматически после merge.

## Применимость по типу репо

- **Сервис:** полный цикл выше (`<hub>/BACKLOG` → feat в сервис-репо → гейт →
  merge → rc → тест). Драйвен программным бэклогом хаба (своего `BACKLOG` не
  несёт).
- **Интерфейс и автономный компонент:** не содержат собственного `BACKLOG` (`docs/refs/LAYOUT.md`),
  но **задачи** для них живут в том же `<hub>/BACKLOG.md` (тег `[<interface>/…]`/
  `[<stub>/…]`). Работа в эти репо приходит изменениями `COMPOSITION`/`CONVENTIONS`
  хаба и ADR — задача формулируется в бэклоге и исполняется агентом по тому же
  гейту (применимость инвариантов — `docs/refs/VERIFICATION.md` → *Применимость
  инвариантов по типу репо*). Баг из теста/прода — в `<hub>/BACKLOG.md`.
