# Деплой (референс)

Сервис деплоится **контейнером** со своим `Dockerfile` в корне репо.
`docker-compose.yml` в корне — **только для локальной разработки**: поднимает
брокер и этот сервис. Системный compose (все сервисы программы вместе)
живёт в **хабе** (или отдельном деплой-репо) и ссылается на собранные образы
сервисов — не здесь.

> Это референс (факт «как устроено»). Процедура локального запуска —
> пошагово в `docs/guide/50-deploy.md`.

## Структура docker-compose.yml (локальная разработка)

```yaml
services:
  broker:              # один на систему (Kafka / Redpanda / NATS)
    image: <broker-image>
    ports: ["9092:9092"]
    volumes: ["broker-data:/data"]

  <service>:            # этот сервис — контейнер из Dockerfile в корне
    build: { context: . }
    depends_on: [broker]
    env_file: .env
    # внешние порты — только если сервис принимает трафик снаружи

volumes:
  broker-data:
```

Правила:

- `build.context: .` — Dockerfile в корне репо.
- `depends_on: [broker]` — сервис читает/пишет в брокер.
- Переменные — из `.env` / `.env.example`; секрета в репо не держать.
- Внешние порты открывать осознанно.
- Это **локальный** compose; production/системный compose — в хабе.

## Dockerfile сервиса

Собирается из команд `build` выбранного стека (`docs/refs/STACKS.md`). Общие
требования:

- Мультистадийная сборка: stage сборки → тонкий runtime-образ.
- В runtime-образе — только артефакт `build` и runtime-зависимости.
- Не запускать от root, где позволяет стек.
- Healthcheck, если сервис его предоставляет.

## Переменные окружения

`/.env.example` — шаблон, коммитится. `/.env` — реальные значения, в
`.gitignore`. Брокер-адрес и per-сервис конфигурация. Имена ключей —
единственный источник в `.env.example`.

## Сборка и проверка перед деплоем

- Прогнать `lint / test / build` (команды — `docs/refs/STACKS.md`).
- `docker compose build` — убедиться, что образ собирается.
- `docker compose config` — проверить валидность compose.
- Собранный образ далее подхватывается системным compose в хабе.

## Интерфейс (frontend)

Интерфейс-репо деплоится **статикой**, не контейнером-сервисом. Брокера в его
окружении **нет**.

- **Сборка:** `vite build` → `dist/` (статические файлы).
- **Раздача:** nginx/CDN. (Опц.) `docker-compose.yml` с nginx-сервисом,
  раздающим `dist/` — только для локальной/preview-проверки собранной статики.
- **Локальная разработка:** `pnpm dev` (Vite dev-сервер), **не** `docker
  compose up` с брокером. Брокер интерфейсу не нужен.
- **Переменные:** `VITE_API_*` (URL потребляемых presentation-эндпоинтов
  сервисов) — в `.env` / `.env.example`. URL эндпоинтов — источник имён ключей.
- **Dockerfile** (если есть): мультистадийный — stage сборки (`pnpm build`) →
  тонкий runtime (nginx + `dist/`), не от root.
- **Проверка перед деплоем:** `pnpm lint && tsc --noEmit && pnpm test &&
  pnpm build`; убедиться, что `dist/` собрался.

> Интерфейс зовёт presentation-эндпоинты сервисов (`docs/refs/COMMUNICATION.md`
> → *Клиентский край*); какие именно — зафиксированы в `docs/ARCHITECTURE.md`
> интерфейса.
