# Общение микросервисов (референс)

**Сервисы обмениваются данными только через брокер.** В системе используется
один брокер: Kafka, Redpanda или NATS. Прямые вызовы между сервисами по HTTP или
RPC, а также общая база данных для нескольких сервисов запрещены. Запрет
распространяется и на связь `gateway → сервис`. Формат сообщений задаётся в
`CONVENTIONS.md` хаба; все сервисы обязаны использовать этот формат.

**Сервис-шлюз** (`gateway`) — единственный сервис, который предоставляет
клиентский HTTP- и WebSocket-API. Он создаётся из `skeletons/service/` и
подчиняется всем требованиям к обычному сервису, включая наличие собственных
`ARCHITECTURE`, `MODULE` и `SPEC`. Шлюз читает темы других сервисов, строит
проекции для чтения, передаёт обновления в браузер и публикует команды оператора
в брокер. Если в системе есть пользовательский интерфейс, должен существовать
ровно один сервис-шлюз. Системе без пользовательских интерфейсов шлюз не нужен.
Остальные сервисы не предоставляют клиентский API.

**Пользовательский интерфейс** не является сервисом и не подключается к брокеру.
Он обращается только к сервису-шлюзу по HTTP или WebSocket. Прямые обращения к
остальным сервисам запрещены. Связь `интерфейс → gateway` является клиентским
соединением и не нарушает запрет на прямой обмен между сервисами.

**Имитатор цели** (`stub`) — самостоятельная программа, которая не участвует в
обмене сообщениями системы. Подробная модель приведена ниже.

> Референс (факт «как система общается»). Топики конкретного сервиса —
> `docs/guide/10-architecture.md` → *Брокер*. Модель верификации контрактов —
> `docs/refs/VERIFICATION.md`. Скелет `CONVENTIONS.md` —
> `skeletons/hub/CONVENTIONS.md`; скелет интерфейса —
> `skeletons/interface/`; шаблон имитатора цели — `skeletons/stub/`.

## Брокер

- **Один брокер на систему** (Kafka / Redpanda / NATS — зафиксируй в хабе).
- Сервис — **клиент брокера**: публикует и/или читает топики.
- Адрес для локальной разработки — из `docker-compose.yml` сервиса (сервис
  `broker`); для системы — из системного compose хаба.

## Конверт сообщения

- Формат сообщений — из хаба `CONVENTIONS.md`. **Не изобретать свой формат в
  спеках сервисов.**

## Топики

Топики конкретного сервиса (publish/consume) — в `docs/ARCHITECTURE.md` →
*Брокер* (таблица: топик / направление / назначение). Спеки модулей ссылаются
на эти топики, не дублируют их.

## Границы

- Прямая **service-to-service** связность в обход брокера (HTTP/RPC/общая БД
  между сервисами, включая `gateway → сервис`) — **запрещена**. Это выход за
  границу одного сервиса; решается в хабе/ADR. gateway берёт данные прочих
  сервисов из брокера (consume топиков), не по HTTP.
- Соединение **пользовательский интерфейс → сервис-шлюз** по HTTP или WebSocket
  разрешено как клиентское. Точки API документируются в разделе
  *Доверительная граница* файла `ARCHITECTURE` сервиса-шлюза. Другие сервисы не
  предоставляют клиентский API.
- Доверительная граница сервиса (где проходит, гарантии,
  подпись/аутентификация; для gateway — presentation-эндпоинты для интерфейсов)
  — в `docs/ARCHITECTURE.md` → *Доверительная граница*.

## Сервис-шлюз

- Сервис-шлюз является обычным клиентом брокера, использует один технологический
  стек, развёртывается в контейнере и имеет собственные `ARCHITECTURE`, `MODULE`
  и `SPEC`. Его отличие — исключительное право предоставлять клиентский API.
  Сервис создаётся из `skeletons/service/`, а его роль фиксируется в
  `COMPOSITION` хаба.
- **Ровно один** на систему, если в программе есть хотя бы один интерфейс.
  Программа без интерфейсов (чистый бэкенд / stub-only cyber-диапазон) gateway
  не требует.
- gateway **потребляет** топики прочих сервисов и держит (при необходимости)
  локальную читающую модель/проекции, чтобы отвечать на GET и пушить WS в
  браузер; **публикует** command-топики для действий оператора. Данные до
  браузера — производные от envelope: gateway трансформирует события в
  HTTP/WS-ответы. Своего envelope для браузера gateway не изобретает — он
  производный от `CONVENTIONS`.
- Клиентские точки HTTP- и WebSocket-API существуют только в сервисе-шлюзе и
  документируются в разделе *Доверительная граница* его `ARCHITECTURE.md`.
  Данные остальных сервисов поступают в шлюз через темы брокера.
- Версия клиентского API задаётся сервисом-шлюзом в URL или заголовке, например
  `/v1/...`. Пользовательский интерфейс фиксирует основную версию. Агент
  проверяет, что заявленные обращения соответствуют документированному API.
- Сеть полностью брокер-опосредована: `интерфейс → gateway (HTTP/WS) → брокер →
  сервисы`. `gateway → сервис` идёт через брокер (consume/publish топиков), не
  через HTTP — `broker-only` сохраняется дословно.
- gateway перечислен в `COMPOSITION` хаба (секция «Сервисы», помечен как
  browser-facing).

## Пользовательский интерфейс

- Пользовательский интерфейс является клиентом и не подключается к брокеру. Он
  обращается только к HTTP- и WebSocket-API сервиса-шлюза, документированному в
  `ARCHITECTURE` шлюза. Прямые обращения к другим сервисам запрещены.
- Данные, доходящие до браузера, — производные от envelope: gateway
  трансформирует события в HTTP/WS-ответы. Свой envelope интерфейс не
  изобретает.
- Версионирование presentation-API — **на gateway** (per-service больше нет);
  интерфейс фиксирует major. Это **conformance** (agent, неблокирует) —
  заявленные интерфейсом вызовы соответствуют документированным эндпоинтам
  gateway.
- Интерфейс перечислен в `COMPOSITION` хаба (секция «Интерфейсы»).

## Имитатор цели

Имитатор цели выполняется как самостоятельная программа и не подключается к
брокеру: он не публикует и не читает темы, не предоставляет клиентский API и не
вызывает сервисы. Поле `form` задаёт способ поставки: контейнер, программа
командной строки или иной исполняемый артефакт. Требования `MODULE` и `SPEC` к
имитатору не применяются. Если у имитатора есть наблюдаемая сетевая, узловая или
процессная поверхность, сборщик контролирует её по отдельному каналу и публикует
результаты в брокер. Связь `stub → collector` фиксируется в `COMPOSITION`, а не
в `CONVENTIONS`.

- Доверительная граница stub'а — в `docs/ARCHITECTURE.md` stub-репо →
  *Доверительная граница*.
- Stub перечислен в `COMPOSITION` хаба. Хаб-контракт к stub **не применяется**
  (нет потребления envelope).

## Связь с верификацией

Ребро `хаб → сервис` проверяет соответствие хабу (`CONVENTIONS`). Ребро
`gateway → интерфейс` (presentation) — agent-conformance: заявленные
интерфейсом вызовы соответствуют `ARCHITECTURE` **gateway** (единственного
browser-facing surface). Потребление gateway'ем топиков прочих сервисов
проверяет ребро `хаб → gateway` (как у любого сервиса: топики соответствуют
`COMPOSITION`). Ребро `хаб → stub` (структурное соответствие + реестр целей в
`COMPOSITION`) и `методология → stub` (репо-тип) — conformance; хаб-контракт
к stub неприменим (он не потребляет envelope).
Проверку соответствия выполняет агент; отдельное заключение не блокирует
изменения. Воспроизводимые проверки CI блокируют изменения при ошибке. Полная
модель описана в `docs/refs/VERIFICATION.md`.
