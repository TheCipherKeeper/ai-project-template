# Общение микросервисов (референс)

**Сервисы** общаются **только через брокер** (один на систему: Kafka / Redpanda
/ NATS). Прямые **service-to-service** вызовы (HTTP/RPC/общая БД между
сервисами, включая `gateway → сервис`) в обход брокера запрещены. Формат
сообщений (event envelope) задаёт **хаб** в `CONVENTIONS.md`, сервисы его
потребляют.

**gateway-сервис** — единственный browser-facing surface программы: обычный
сервис (broker-клиент, инстанциация из `skeletons/service/`, свой
`ARCHITECTURE`/`MODULE`/`SPEC`) с **канонической ролью**. Потребляет топики
прочих сервисов (читает/проекции/WS-пуш в браузер) и публикует command-топики
(действия оператора); экспонирует HTTP/WS presentation-эндпоинты для интерфейсов.
Ровно один на систему, если есть хотя бы один интерфейс (программа без интерфейсов
gateway не требует). Presentation-эндпоинты для интерфейсов живут **только** на
нём — прочие сервисы их не держат, их клиентский край = топики брокера. Так сеть
полностью брокер-опосредована: `интерфейс → gateway (HTTP/WS) → брокер → сервисы`.

**Интерфейсы** — не брокер-клиенты и не peer-сервисы. Это клиенты на
доверительной границе: зовут **только gateway-сервис** по HTTP/WS
presentation-эндпоинтам (модель — *gateway-сервис* и *Клиентский край* ниже). Прямо
в прочие сервисы интерфейс не ходит. Это не нарушает broker-only — то правило
управляет service↔service; `интерфейс → gateway` — клиентский край, не
service-to-service.

**Stub-таргеты** — standalone-программы, не участник общения (модель —
*Stub-таргет* ниже).

> Референс (факт «как система общается»). Топики конкретного сервиса —
> `docs/guide/10-architecture.md` → *Брокер*. Модель верификации контрактов —
> `docs/refs/VERIFICATION.md`. Скелет `CONVENTIONS.md` —
> `skeletons/hub/CONVENTIONS.md`; скелет интерфейса —
> `skeletons/interface/`; скелет stub-таргета — `skeletons/stub/`.

## Брокер

- **Один брокер на систему** (Kafka / Redpanda / NATS — зафиксируй в хабе).
- Сервис — **клиент брокера**: публикует и/или читает топики.
- Адрес для локальной разработки — из `docker-compose.yml` сервиса (сервис
  `broker`); для системы — из системного compose хаба.

## Event envelope

- Формат сообщений — из хаба `CONVENTIONS.md`. **Не изобретать свой формат в
  спеках сервисов.**

## Топики

Топики конкретного сервиса (publish/consume) — в `docs/ARCHITECTURE.md` →
*Брокер* (таблица: топик / направление / назначение). Спеки модулей ссылаются
на эти топики, не дублируют их.

## Границы

- Прямая **service-to-service** связность в обход брокера (HTTP/RPC/общая БД
  между сервисами, включая `gateway → сервис`) — **запрещена**. Это выход за
  границу одного сервиса; решается в хабе/ADR. gateway берёт данные прочих
  сервисов из брокера (consume топиков), не по HTTP.
- **Интерфейс → gateway-сервис** по HTTP/WS presentation-эндпоинтам — разрешено
  (клиент на границе); документируется в `ARCHITECTURE` **gateway** →
  *Доверительная граница*. **Только** gateway держит browser-facing
  presentation-эндпоинты; прочие сервисы — нет.
- Доверительная граница сервиса (где проходит, гарантии,
  подпись/аутентификация; для gateway — presentation-эндпоинты для интерфейсов)
  — в `docs/ARCHITECTURE.md` → *Доверительная граница*.

## gateway-сервис (клиентский край)

- gateway — **обычный сервис** (broker-клиент, один стек, контейнер, свой
  `ARCHITECTURE`/`MODULE`/`SPEC`), но с **канонической ролью**: единственный
  browser-facing surface программы. Инстанциация из `skeletons/service/`;
  назначение роли фиксируется в `COMPOSITION` хаба.
- **Ровно один** на систему, если в программе есть хотя бы один интерфейс.
  Программа без интерфейсов (чистый бэкенд / stub-only cyber-диапазон) gateway
  не требует.
- gateway **потребляет** топики прочих сервисов и держит (при необходимости)
  локальную читающую модель/проекции, чтобы отвечать на GET и пушить WS в
  браузер; **публикует** command-топики для действий оператора. Данные до
  браузера — производные от envelope: gateway трансформирует события в
  HTTP/WS-ответы. Своего envelope для браузера gateway не изобретает — он
  производный от `CONVENTIONS`.
- **Presentation-эндпоинты для интерфейсов (HTTP/WS)** живут **только** на
  gateway и документируются в его `ARCHITECTURE.md` → *Доверительная граница*.
  Прочие сервисы browser-facing presentation не держат — их клиентский край
  достигается через топики, потребляемые gateway.
- Версионирование presentation-API — **на gateway** (одно место, не
  per-service): версия в URL/заголовке (`/v1/...`); интерфейс фиксирует major.
  Это **conformance** (agent, неблокирует) — заявленные интерфейсом вызовы
  соответствуют документированным эндпоинтам gateway.
- Сеть полностью брокер-опосредована: `интерфейс → gateway (HTTP/WS) → брокер →
  сервисы`. `gateway → сервис` идёт через брокер (consume/publish топиков), не
  через HTTP — `broker-only` сохраняется дословно.
- gateway перечислен в `COMPOSITION` хаба (секция «Сервисы», помечен как
  browser-facing).

## Клиентский край (interface)

- Интерфейс — клиент, не участник брокера. Зовёт **только** presentation-эндпоинты
  **gateway-сервиса** (HTTP/WS), документированные в `ARCHITECTURE` gateway. Прямо
  в прочие сервисы не ходит — одного URL/CORS/auth достаточно.
- Данные, доходящие до браузера, — производные от envelope: gateway
  трансформирует события в HTTP/WS-ответы. Свой envelope интерфейс не
  изобретает.
- Версионирование presentation-API — **на gateway** (per-service больше нет);
  интерфейс фиксирует major. Это **conformance** (agent, неблокирует) —
  заявленные интерфейсом вызовы соответствуют документированным эндпоинтам
  gateway.
- Интерфейс перечислен в `COMPOSITION` хаба (секция «Интерфейсы»).

## Stub-таргет (standalone-программа)

Stub — standalone-программа, не участник брокера: не publish/consume топики, не
имеет presentation-эндпоинтов, никого не зовёт. Форма (`form`: контейнер / CLI / …) —
параметр из control-plane; деплой по форме (контейнер — `Dockerfile`, CLI —
артефакт). `MODULE`/`SPEC` к stub не применяются (он параметризуется, не
юзкейсами). Если у stub'а есть наблюдаемая поверхность (сеть/хост/процесс) —
collector наблюдает её **out-of-band** (мимо брокера) и публикует события в брокер;
без поверхности — просто поставляемая standalone-программа. Это не нарушает
broker-only: stub не участник общения, а наблюдение — не брокерный контракт
stub'а (ребро `stub → collector` фиксируется в `COMPOSITION`, не в `CONVENTIONS`).

- Доверительная граница stub'а — в `docs/ARCHITECTURE.md` stub-репо →
  *Доверительная граница*.
- Stub перечислен в `COMPOSITION` хаба. Хаб-контракт к stub **не применяется**
  (нет потребления envelope).

## Связь с верификацией

Ребро `хаб → сервис` проверяет соответствие хабу (`CONVENTIONS`). Ребро
`gateway → интерфейс` (presentation) — agent-conformance: заявленные
интерфейсом вызовы соответствуют `ARCHITECTURE` **gateway** (единственного
browser-facing surface). Потребление gateway'ем топиков прочих сервисов
проверяет ребро `хаб → gateway` (как у любого сервиса: топики соответствуют
`COMPOSITION`). Ребро `хаб → stub` (структурное соответствие + реестр целей в
`COMPOSITION`) и `методология → stub` (репо-тип) — conformance; хаб-контракт
к stub неприменим (он не потребляет envelope).
Conformance — агент, неблокирует; behavioral — детерминированный CI, блокирует.
Полная модель — `docs/refs/VERIFICATION.md`.
