# Общение микросервисов (референс)

**Сервисы** общаются **только через брокер** (один на систему: Kafka / Redpanda
/ NATS). Прямые **service-to-service** вызовы (HTTP/RPC/общая БД между
сервисами) в обход брокера запрещены. Формат сообщений (event envelope) задаёт
**хаб** в `CONVENTIONS.md`, сервисы его потребляют на пиннённой версии.

**Интерфейсы** — не брокер-клиенты и не peer-сервисы. Это клиенты на
доверительной границе: зовут сервисы по **HTTP/WS presentation-эндпоинтам**,
которые сервис документирует у себя в `ARCHITECTURE.md` → *Доверительная
граница*. Это не нарушает broker-only — то правило управляет service↔service.

**Stub-таргеты** — не брокер-клиенты и не peer-сервисы. Это пассивные цели
(deployed-контейнер с реальными сетевыми поверхностями), параметризуемые
дескриптором из manage и **наблюдаемые collector'ом out-of-band**. Stub не
publish/consume топики и не имеет presentation-эндпоинтов — он не участник
общения, а объект наблюдения. Это не нарушает broker-only.

> Референс (факт «как система общается»). Топики конкретного сервиса —
> `docs/guide/10-architecture.md` → *Брокер*. Модель верификации контрактов —
> `docs/refs/VERIFICATION.md`. Скелет `CONVENTIONS.md` —
> `skeletons/hub/CONVENTIONS.md`; скелет интерфейса —
> `skeletons/interface/`; скелет stub-таргета — `skeletons/stub/`.

## Брокер

- **Один брокер на систему** (Kafka / Redpanda / NATS — зафиксируй в хабе).
- Сервис — **клиент брокера**: публикует и/или читает топики.
- Адрес для локальной разработки — из `docker-compose.yml` сервиса (сервис
  `broker`); для системы — из системного compose хаба.

## Event envelope

- Формат сообщений — из хаба `CONVENTIONS.md`. **Не изобретать свой формат в
  спеках сервисов.**
- Хаб экспонирует контракты с версиями: `CONVENTIONS@v1`, `@v2`, …
- Сервис **пинит** версию в своей конфигурации/манифесте
  (`docs/ARCHITECTURE.md` → *Контракты хаба*); гейт проверяет против пина, не
  HEAD. Механика версионирования и миграции (`@v2` не красит `@v1`, бамп пина —
  отдельным PR сервиса) — `docs/refs/VERIFICATION.md` → *Версионирование
  контрактов*.

## Топики

Топики конкретного сервиса (publish/consume) — в `docs/ARCHITECTURE.md` →
*Брокер* (таблица: топик / направление / назначение). Спеки модулей ссылаются
на эти топики, не дублируют их.

## Границы

- Прямая **service-to-service** связность в обход брокера (HTTP/RPC/общая БД
  между сервисами) — **запрещена**. Это выход за границу одного сервиса;
  решается в хабе/ADR.
- **Интерфейс → сервис** по HTTP/WS presentation-эндпоинтам — разрешено
  (клиент на границе); документируется в сервис `ARCHITECTURE` →
  *Доверительная граница*.
- Доверительная граница сервиса (где проходит, гарантии,
  подпись/аутентификация, presentation-эндпоинты для интерфейсов) — в
  `docs/ARCHITECTURE.md` → *Доверительная граница*.

## Клиентский край (interface)

- Интерфейс — клиент, не участник брокера. Зовёт presentation-эндпоинты
  сервисов (HTTP/WS), документированные в `ARCHITECTURE` сервисов.
- Данные, доходящие до браузера, — производные от envelope: сервис
  трансформирует события в HTTP/WS-ответы. Свой envelope интерфейс не
  изобретает.
- Версионирование presentation-API — **per-service**: рекомендуется версия в
  URL/заголовке (`/v1/...`); интерфейс фиксирует major. Это **conformance**
  (agent, неблокирует), не behavioral — проверяет, что заявленные интерфейсом
  вызовы соответствуют документированным эндпоинтам сервисов.
- Интерфейс перечислен в `COMPOSITION` хаба (секция «Интерфейсы»).

## Stub-таргет (passive target)

- Stub — пассивная цель, **не участник брокера**: не publish/consume топики, не
  имеет presentation-эндпоинтов, не зовёт чужие сервисы/БД. Сам никого не вызывает.
- Stub **наблюдается collector'ом out-of-band**: collector сканирует/слушит
  сетевые поверхности stub'а (raw TCP/SSH/HTTP-баннер) и публикует подписанные
  события наблюдения в брокер. Это ребро `stub → collector` реализуется как
  **операционный факт collector'а** (он активная сторона), а не как брокерный
  контракт stub'а — stub про брокер ничего не знает. Фиксируется в `COMPOSITION`
  хаба (какие collector'ы какие stub'ы наблюдают), не в `CONVENTIONS@vN`.
- Stub **параметризуется дескриптором** из control-plane (manage): `kind`, `ports`,
  `software`, `banner`, `runtime_kind`. Дескриптор — не брокерное сообщение;
  модель `runtime_kind` (вкл. `lite`-stub) — решение хаба (ADR).
- Доверительная граница stub'а (где проходит, что по какую сторону, гарантии,
  подпись наблюдений collector'а) — в `docs/ARCHITECTURE.md` stub-репо →
  *Доверительная граница*.
- Stub перечислен в `COMPOSITION` хаба (секция «Stub-таргеты»). `CONVENTIONS@vN`
  к stub **не применяется** (нет потребления envelope) — пин в его `ARCHITECTURE`
  не ставится.

## Связь с верификацией

Ребро `хаб → сервис` проверяет соответствие хабу на пиннённой версии
контрактов; без пина ребро «вниз» нерабочее. Ребро `сервис → интерфейс`
(presentation) — agent-conformance: заявленные интерфейсом вызовы
соответствуют `ARCHITECTURE` сервисов. Ребро `хаб → stub` (структурное
соответствие + реестр целей в `COMPOSITION`) и `методология → stub`
(репо-тип) — conformance, без пина контракта (`CONVENTIONS@vN` N/A для stub).
Conformance — агент, неблокирует; behavioral — детерминированный CI, блокирует.
Полная модель — `docs/refs/VERIFICATION.md`.
