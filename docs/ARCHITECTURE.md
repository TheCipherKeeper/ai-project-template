# Архитектурный канон

## Принцип минимальной распределённости

Система не обязана начинаться набором микросервисов. По умолчанию новая
функциональность становится модулем существующего сервиса. Новый сервис создаётся,
только если есть хотя бы одна независимая граница:

- владение данными и транзакциями;
- доверие или безопасность;
- масштабирование и профиль нагрузки;
- поставка и жизненный цикл;
- отказоустойчивость или технологическая изоляция.

Удобство разложения каталогов само по себе не является причиной нового сервиса.

## Топология

Хаб хранит `COMPOSITION.md`, `CONVENTIONS.md`, `BACKLOG.md`, системный compose и
ADR. Сервисы, интерфейсы и автономные компоненты живут в отдельных репозиториях
только тогда, когда их независимая поставка действительно нужна. Они ссылаются
на версию методологии, но не копируют её документы.

Сервис использует один основной язык. Внутри рекомендуется структура по
вариантам использования с явными границами domain, application и adapters;
небольшой модуль не обязан имитировать все слои пустыми каталогами.

## Коммуникация

- Интерфейс обращается к browser-facing gateway по HTTP/WS.
- Асинхронные межсервисные события проходят через выбранный брокер и общий
  versioned envelope.
- Синхронный service-to-service вызов допустим, если запрос по природе
  синхронный, имеет timeout, ограниченные retries, наблюдаемость и не создаёт
  цикл зависимостей. Решение фиксируется в архитектуре сервиса.
- Контракт имеет владельца, версию, схему и consumer/contract tests.
- Секреты не передаются в событиях, репозиториях или evidence.

## Рабочие артефакты

- `docs/ARCHITECTURE.md` — роль компонента, границы, зависимости и потоки.
- `docs/specs/<module>.md` — наблюдаемое поведение и контракты сложного модуля.
- `COMPOSITION.md` — состав программы и версии поставляемых компонентов.
- `CONVENTIONS.md` — только общесистемные соглашения.
- `adr/` — труднообратимые решения с несколькими реальными альтернативами.

Спека не обязательна для тривиального модуля, если контракт и поведение полностью
выражены кодом, типами и тестами. Она обязательна для публичного контракта,
нетривиальной доменной логики или поведения, которое нельзя быстро восстановить
из реализации.

## Изменение архитектуры

Агент должен:

1. показать существующую и предлагаемую границу;
2. назвать получаемое преимущество и операционную цену;
3. проверить обратную совместимость контрактов и данных;
4. обновить архитектуру, тесты контрактов и compose;
5. создать ADR только для труднообратимого общесистемного решения.
