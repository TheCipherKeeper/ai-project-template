# Архитектурный канон

## Авторство высокоуровневой архитектуры

Человек единолично определяет необходимые репозитории, их количество, назначение
и высокоуровневые границы системы. Решение фиксируется в `COMPOSITION.md` и при
необходимости в задаче или ADR. Агент не создаёт и не удаляет репозитории, не
меняет их назначение и не переносит ответственность между ними без явного
решения человека.

При выборе границ человеку рекомендуется избегать обязательной распределённости.
Новый сервис обычно оправдан независимой границей:

- владение данными и транзакциями;
- доверие или безопасность;
- масштабирование и профиль нагрузки;
- поставка и жизненный цикл;
- отказоустойчивость или технологическая изоляция.

Удобство разложения каталогов само по себе не является причиной нового сервиса.
Если в ходе реализации заданных границ недостаточно, агент описывает потребность
и ставит `needs-input`.

## Топология

Хаб хранит `COMPOSITION.md`, `CONVENTIONS.md`, `BACKLOG.md`, системный compose и
ADR. Сервисы, интерфейсы и автономные компоненты живут в отдельных репозиториях
только тогда, когда их независимая поставка действительно нужна. Они ссылаются
на версию методологии, но не копируют её документы.

Сервис использует только Python, Go или Rust и обязательную гексагональную
архитектуру. Каждый самостоятельный модуль имеет все каталоги `domain`,
`application`, `ports/inbound`, `ports/outbound`, `adapters/inbound` и
`adapters/outbound`. До появления кода пустой каталог содержит краткий `README.md`;
фиктивные интерфейсы, реализации и TODO ради заполнения структуры запрещены.
Модулем считается каждая строка таблицы модулей в `docs/ARCHITECTURE.md`; весь
прикладной код принадлежит ровно одному такому модулю, кроме composition root.
Имя сервиса, язык и тот же упорядоченный список модулей фиксируются в
`.methodology.yml` полями `service_name`, `service_language` и `service_modules`;
расхождение с кодом блокирует гейт.

### Слои и зависимости сервиса

| Часть | Содержит | Может зависеть от |
|---|---|---|
| `domain` | сущности, value objects, доменные правила и события | только стандартная библиотека |
| `ports/inbound` | контракты возможностей модуля | `domain` |
| `ports/outbound` | контракты нужных модулю хранилищ и интеграций | `domain` |
| `application` | реализации inbound ports и варианты использования | `domain`, оба вида `ports` |
| `adapters/inbound` | HTTP, broker consumers, CLI, преобразование входа | `domain`, `ports/inbound` |
| `adapters/outbound` | реализации outbound ports: БД, broker producers, внешние системы | `domain`, `ports/outbound` |

Любая зависимость, не указанная в таблице, запрещена. Фреймворки транспорта,
ORM, брокера и хранения допустимы только в `adapters`; их типы не пересекают
порты. Сигнатуры портов используют только типы `domain` или определённые рядом с
портом транспортно-независимые DTO. Inbound port реализует `application`,
outbound port — соответствующий outbound adapter. Адаптеры не вызывают друг
друга. Между модулями разрешён вызов
только через inbound port целевого модуля; прямой импорт его `domain`,
`application` или `adapters` запрещён.

Composition root только создаёт реализации и связывает порты с адаптерами;
бизнес-логика в нём запрещена. Его единственное расположение зависит от языка:

| Язык | Корень модуля | Composition root |
|---|---|---|
| Python | `src/<service>/<module>/` | `src/<service>/bootstrap.py` |
| Go | `internal/<module>/` | `cmd/<service>/main.go` |
| Rust | `src/<module>/`; `mod.rs` в корне модуля, `ports`, `adapters` и каждом их обязательном каталоге | `src/main.rs` |

Иные названия слоёв и альтернативное расположение composition root запрещены.
Composition root содержит только импорты, создание и связывание объектов и один
запуск inbound adapter; определения типов, вспомогательных функций и управляющая
логика запрещены. В Rust-модулях запрещены вложенные групповые `use`.
Тесты повторяют модульные границы в `tests/unit`, `tests/integration` и
`tests/contract`; неприменимый вид тестов сохраняет краткий `README.md` с причиной.

## Коммуникация

- Интерфейс обращается к клиентскому сервису-шлюзу по HTTP/WS.
- Асинхронные межсервисные события проходят через выбранный брокер и общий
  versioned envelope.
- Service-to-service взаимодействие выполняется только через брокер. Прямые
  HTTP, gRPC, WebSocket и иные синхронные вызовы между сервисами запрещены.
- HTTP/WS допустим только на клиентской границе `интерфейс → клиентский
  сервис-шлюз`; сервис-шлюз взаимодействует с другими сервисами через брокер.
- Контракт имеет владельца, версию, схему, проверки потребителя и проверки контракта.
- Секреты не передаются в событиях, репозиториях или свидетельствах выполнения.

Для event-контракта обязательны: producer и consumers, payload schema,
partition/ordering key, delivery semantics, idempotency key, retry/DLQ,
максимальный размер, retention, PII-класс и compatibility policy. HTTP/WS
описывается OpenAPI, события — AsyncAPI и JSON Schema/Protobuf/Avro.

Breaking contract требует отдельной задачи, ADR, `BREAKING CHANGE:` и плана
миграции consumers. Совместимые consumers разворачиваются раньше producer,
начинающего публиковать новую форму. Удаление поля выполняется только после
периода совместимости; проверки контрактов блокируют слияние.

## Рабочие артефакты

- `docs/ARCHITECTURE.md` — роль компонента, границы, зависимости и потоки.
- `docs/specs/<module>.md` — ответственность, наблюдаемое поведение, инварианты,
  ошибки, зависимости и проверка каждого самостоятельного модуля.
- `COMPOSITION.md` — состав программы и версии поставляемых компонентов.
- `CONVENTIONS.md` — только общесистемные соглашения.
- `adr/` — труднообратимые решения с несколькими реальными альтернативами.

Спека обязательна для каждого самостоятельного модуля, но для простого модуля
может быть краткой. Она фиксирует замысел и ограничения, а не пересказывает код;
публичные контракты, состояния и нетривиальные правила описываются полностью.

## Диаграммы

Все диаграммы в Markdown во всех репозиториях — методологии, хабе, сервисах,
интерфейсах и автономных компонентах — оформляются только блоками `mermaid`.
Это относится к схемам архитектуры, зависимостей, потоков данных, состояний,
последовательностей, развёртывания и рабочего процесса.

ASCII-псевдографика, текстовые схемы со стрелками, PlantUML, Graphviz, D2 и
другие отдельные языки диаграмм запрещены. Блоки команд, журналов и примеров
данных диаграммами не считаются и должны иметь точное обозначение языка, например
`console`, `json` или `yaml`. Безымянный блок или блок `text` со стрелками либо
псевдографикой считается диаграммой. Изображение допустимо как иллюстрация,
но не заменяет редактируемую Mermaid-диаграмму, если изображение описывает
архитектуру, процесс или связи между компонентами.

## Реализация архитектурного решения

После того как человек задал высокоуровневое изменение, агент должен:

1. показать существующую и предлагаемую границу;
2. назвать получаемое преимущество и операционную цену;
3. проверить обратную совместимость контрактов и данных;
4. обновить архитектуру, тесты контрактов и compose;
5. создать ADR только для труднообратимого общесистемного решения.

Агент вправе уточнять внутреннюю структуру кода и техническую реализацию
контрактов, но не изменять количество, назначение или ответственность
репозиториев.
