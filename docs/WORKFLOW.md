# Рабочий цикл одной задачи

## 1. Выбрать задачу

Агент берёт первый пункт `[ ] ready` из `<hub>/BACKLOG.md`. Другую задачу он не
начинает до завершения или диагностической остановки текущей.

Минимально готовая задача содержит:

- устойчивый `TASK-NNNN`;
- цель и наблюдаемое поведение;
- проверяемые критерии готовности;
- существенные ограничения или явное «нет».

Для каждой записи `BACKLOG.md` хаб содержит `.tasks/TASK-NNNN.json` по
`schemas/task.schema.json`. Человек задаёт intent в backlog, агент синхронно
отражает его без изменения смысла в машинной записи; verifier сверяет ID и
статус обоих представлений.

Если отсутствует продуктовое решение, агент не угадывает его, а ставит один из
статусов `needs-input`, `blocked-external`, `automation-failed` или
`retry-exhausted` и пишет короткую диагностику. Технические детали агент вправе
определить самостоятельно в пределах архитектурного канона и заданных человеком
репозиторных границ.

## 2. Квалифицировать изменение

До кода исполнитель фиксирует в PR или рабочем отчёте:

- целевые репозитории и артефакты;
- проверяемое соответствие каждого критерия результату;
- применимые триггеры: контракт, данные, безопасность, инфраструктура,
  архитектура;
- план реализации и проверки;
- способ безопасного отката для поставляемого изменения.

Исполнитель назначает риск и предел автономности по `REFERENCE.md`. Другой агент
может сделать классификацию строже. Понижение риска или расширение автономности
разрешает только человек с записью причины в PR.

Триггер определяет дополнительные проверки. Полная матрица находится в
[`REFERENCE.md`](REFERENCE.md).

Если для выполнения требуется создать или удалить репозиторий, изменить его
назначение либо перенести высокоуровневую ответственность между репозиториями,
агент ставит `needs-input`. Такое решение принимает и явно фиксирует человек.

## 3. Реализовать

1. Обновить локальный `main` из `origin/main`, создать
   `feat/TASK-NNNN-<slug>`.
2. Изучить локальный `AGENTS.md`, задачу и только применимые документы.
3. Изменить код, тесты и рабочую документацию одним согласованным набором.
4. Не расширять scope ради необязательного рефакторинга.
5. Сделать Conventional Commit, отправить feature-ветку в `origin` и открыть PR
   в `main`.

Если задача затрагивает несколько репозиториев, агент ведёт один логический
план, но отдельную ветку и PR в каждом репозитории. Порядок merge и совместимость
контрактов указываются в PR.

## 4. Проверить

Сначала выполняются воспроизводимые проверки репозитория:

- policy/verifier методологии;
- format/lint/typecheck;
- tests;
- build;
- дополнительные contract, migration или security checks по триггеру.

Недоступный обязательный гейт — это `automation-failed`, а не зелёный результат.

После зелёного гейта:

- документация и локальный рефакторинг могут идти сразу к merge;
- обычная функциональность получает независимое review;
- сложное поведение получает отдельную проверку сценариев;
- security-sensitive изменение получает отдельный security lens.

Reviewer читает задачу и контракты до реализации. Каждую находку он пытается
опровергнуть воспроизводимым примером. Подтверждённая находка исправляется новым
коммитом с повторным гейтом; неподтверждённая отбрасывается.

Допустимо не более трёх общих циклов `review → fix → recheck` и не более двух
повторений одной категории ошибки. Затем PR остаётся открытым, задача получает
`retry-exhausted`, а человеку передаётся причина и уже проверенные варианты.

## 5. Слить и проверить результат

Если человек явно не запретил публикацию, работа агента не заканчивается на
локальных файлах или локальном commit. После зелёного gate агент самостоятельно
завершает полный Git-цикл:

1. отправляет feature-ветку в `origin`;
2. создаёт PR в удалённый `main` и прикладывает результаты проверок;
3. дожидается обязательных checks и устраняет подтверждённые замечания в
   пределах лимита повторов;
4. выполняет squash merge зелёного PR;
5. убеждается, что merge commit появился в `origin/main`, переключается на
   локальный `main` и синхронизирует его с `origin/main`;
6. проверяет чистое рабочее дерево и сообщает task ID, PR и merge commit.

Прямой push локального commit в `main` не заменяет PR и запрещён. Агент
останавливается до публикации только при явном запрете человека, недостаточных
правах, недоступном remote/CI или обязательном подтверждении по уровню риска; в
этом случае он сообщает точную диагностику и не выдаёт локальный commit за
завершённую задачу.

После squash merge агент собирает артефакт по merge commit SHA и разворачивает
его в test. Затем запускаются readiness, smoke и применимые contract/security
probes.

Задача получает `[x]` только когда зафиксированы:

- task ID, merge commit и версия методологии;
- результаты обязательных проверок и review;
- идентификатор поставленного артефакта;
- зелёные post-deploy проверки.

Завершённая машинная задача обязана иметь `.evidence/TASK-NNNN.json` по
`schemas/evidence.schema.json`; verifier проверяет его связь с задачей,
methodology ref и переданным CI commit.

Если test-проверка не прошла, агент применяет заранее проверенный rollback или
останавливается с точной диагностикой. Новый продуктовый баг человек оформляет
новой задачей.

Агент никогда не меняет acceptance criteria, не скрывает непройденную проверку,
не принимает остаточный риск и не выполняет необратимое действие за пределами
явно разрешённой задачи.

## Особые случаи

- Заданная человеком архитектурная граница или новый сервис — открыть
  [`ARCHITECTURE.md`](ARCHITECTURE.md).
- Миграция, rollback, production или release — открыть
  [`OPERATIONS.md`](OPERATIONS.md).
- Секреты, доступ, сеть, файловая система или недоверенный ввод — обязательный
  security lens.
- Труднообратимое общесистемное решение — ADR в хабе. Обычный рефакторинг ADR не
  требует.
