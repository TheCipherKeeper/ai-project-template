# Заготовки репозиториев

Здесь находятся стартовые наборы файлов, а не четыре части, которые обязательно
нужно создать. Количество репозиториев, их назначение и границы определяет
человек. Выберите только явно нужные типы и скопируйте содержимое соответствующих
каталогов в отдельные репозитории.

## Как выбрать

| Заготовка | Когда использовать | Не использовать как |
|---|---|---|
| [`hub/`](hub/) | единый реестр системы, backlog, общие контракты, системный compose и ADR | место для кода компонентов |
| [`service/`](service/) | независимо поставляемый сервис, участвующий в обмене через брокер | внутренний модуль другого сервиса |
| [`interface/`](interface/) | клиентское приложение, обращающееся к browser-facing gateway | сервис или broker consumer |
| [`standalone/`](standalone/) | независимо поставляемая программа вне сервисного обмена | временная заглушка или скрытый сервис |

Автономный компонент имеет машинный тип `standalone`. Это одно понятие; старое
имя `stub` не используется.

## Как читать одну заготовку

Не нужно читать все файлы подряд. Для выбранного типа:

1. Откройте `README.md`: там назначение, первичная настройка и локальный запуск.
2. Заполните файлы из группы «при создании» в указанном порядке.
3. Прочитайте `AGENTS.md`: это локальные ограничения для исполнителя.
4. Не редактируйте `.tasks/` и `.evidence/` вручную: их создаёт автоматизация.
5. Запустите локальные проверки и средство проверки методологии.

## Общая анатомия

| Артефакт | Владелец факта | Когда меняется |
|---|---|---|
| `README.md` | навигация и команды старта | когда меняется путь разработчика |
| `AGENTS.md` | локальные ограничения и точные команды проверок | вместе с правилами репозитория |
| `.methodology.yml` | машинный тип, закреплённая версия методологии и структурная идентичность | при создании и явном архитектурном изменении |
| `docs/ARCHITECTURE.md` | назначение, границы, контракты и фактическое устройство | вместе с соответствующим изменением |
| `docs/specs/` | наблюдаемое поведение самостоятельных модулей сервиса | вместе с поведением модуля |
| `.env.example` | имена конфигурационных параметров без секретов | вместе с конфигурационным контрактом |
| `Dockerfile`, `docker-compose.yml` | сборка и локальный запуск | вместе со способом поставки |
| `.github/workflows/verify.yml` | обязательная проверка методологии | при изменении общего CI-шаблона |
| `.tasks/`, `.evidence/` | машинные записи задач и выполнения | только автоматикой |

Системные факты хранятся в хабе: состав — в `COMPOSITION.md`, общие контракты —
в `CONVENTIONS.md`, задачи — в `BACKLOG.md`, труднообратимые решения — в `adr/`.
README компонентов на них ссылаются, но не создают вторые реестры.

## Маршрут знакомства

Для первого знакомства достаточно последовательно прочитать
`docs/START.md`, эту страницу, `README.md` выбранной заготовки и её `AGENTS.md`.
`docs/WORKFLOW.md` открывается при выполнении первой задачи, остальные документы
методологии — по условиям из `docs/INDEX.md`.
