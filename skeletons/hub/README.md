# <project> — хаб

Хаб хранит системный контекст программы. Канонический состав находится в
`COMPOSITION.md`, общие контракты — в `CONVENTIONS.md`, задачи — в `BACKLOG.md`.
Кода сервисов, интерфейсов и автономных компонентов здесь нет.

## Первичная настройка

Выполните один раз в указанном порядке:

1. В `.methodology.yml` закрепите точный тег или коммит методологии.
2. В `COMPOSITION.md` перечислите только выбранные человеком репозитории,
   назначение, версии или хеши артефактов и связи между ними.
3. В `CONVENTIONS.md` зафиксируйте выбранный брокер, конверт события и только
   общесистемные соглашения.
4. В `BACKLOG.md` замените пример первой реальной задачей и запустите
   `uv run <methodology-repo>/tools/tasks/sync.py`.
5. Настройте системный `docker-compose.yml` и имена параметров в `.env.example`.
6. Проверьте команды и выходы сборки в `.pipeline.json`.
7. Задайте переменные CI `METHODOLOGY_REPOSITORY`, `METHODOLOGY_VERSION`,
   `AGENT_REVIEWER_LOGIN` и `HUMAN_REVIEWER_LOGIN`; настройте среды `test` и
   защищённую `critical-test`.
8. Укажите лицензию ниже и выполните локальную проверку.

## Карта файлов

| Файл | Назначение | Кто меняет |
|---|---|---|
| `COMPOSITION.md` | единственный реестр компонентов, версий и связей | человек задаёт границы, агент отражает решение |
| `CONVENTIONS.md` | единственный источник общесистемных контрактов | агент в рамках явной задачи |
| `BACKLOG.md` | единственный источник истины о задачах | человек формулирует и упорядочивает |
| `docker-compose.yml`, `.env.example` | системный запуск и контракт конфигурации | агент вместе с поставкой |
| `adr/` | труднообратимые системные решения | агент по архитектурному триггеру |
| `.methodology.yml`, `AGENTS.md` | машинная идентичность и локальные правила | при создании или изменении правил |
| `.pipeline.json`, `.github/workflows/verify.yml` | команды и исполнимый продуктовый конвейер | вместе с инструментами и поставкой |
| `.tasks/`, `.evidence/` | генерируемые машинные записи | только автоматизация |

## Разработка

Задача выполняется по `<methodology-repo>/docs/WORKFLOW.md`. Перед коммитом
запустите средство проверки методологии и применимые проверки из
`<methodology-repo>/docs/REFERENCE.md`. Стабильные версии создаются отдельной
задачей и отмечаются тегами `vX.Y.Z` на `main`.

## Локальный запуск

```bash
cp .env.example .env
docker compose up
```

Файл `.env` не коммитится. Эксплуатационные правила находятся в
`<methodology-repo>/docs/OPERATIONS.md`.
