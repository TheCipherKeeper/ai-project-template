# ADR 0001 — Записывать архитектурные решения

Дата: <YYYY-MM-DD>
Статус: Принято

## Контекст

Нужен способ фиксировать значимые **системные** решения программы и причину их
принятия. Хаб — единый ADR-дом программы: здесь живут решения уровня программы
(выбор брокера, формат конверта события, состав программы). Без истории решений агентам и
новым разработчикам непонятно, почему система
устроена именно так, и есть риск откатить решение, не зная его обоснования.

## Решение

Завести каталог ADR (`adr/`). Каждое решение — отдельный файл с однострочной
нумерацией (`000N-<slug>.md`). Структура: Контекст / Решение / Последствия.
Старые решения помечаем `Superseded`, не удаляем — история должна оставаться
читаемой. Ссылки на ADR — из `COMPOSITION.md` и `CONVENTIONS.md`.

## Последствия

- Появляется история системных решений: видно, что менялось и почему.
- Расхождение доков или спорное системное изменение → повод для нового ADR.
- Затраты: один файл на значимое решение. Окупается читаемостью для агентов.
