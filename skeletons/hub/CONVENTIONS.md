## Конверт события

Общий формат сообщения в теме брокера. Все сервисы публикуют и читают его.

<!-- Заполни поля под программу. Пример ниже — заготовка. -->

```json
{
  "event_type": "<service>.<domain>.<action>",
  "event_id": "<uuid>",
  "occurred_at": "<RFC3339>",
  "producer": "<service>",
  "trace_id": "<uuid>",
  "payload": { }
}
```

| Поле | Тип | Обяз. | Назначение |
|---|---|---|---|
| `event_type` | string | да | тип события; пространство имён = `<service>.<domain>.<action>` |
| `event_id` | uuid | да | идемпотентность: подписчик устраняет дубликаты по этому полю |
| `occurred_at` | RFC3339 | да | когда событие произошло (не когда опубликовано) |
| `producer` | string | да | сервис-источник |
| `trace_id` | uuid | опц. | сквозная трассировка |
| `payload` | object | да | данные события; схема по `event_type` |

## Темы брокера

- Именование: `<domain>.<event>` (например `billing.invoice_created`).
- Сервис владеет темами, в которые публикует события; подписчик читает их.
- Реестр тем по сервисам — `COMPOSITION.md` → *Сервисы*.

## Правила

- Общение — **только через брокер**; прямые межсервисные вызовы
  (включая `сервис-шлюз → сервис`) запрещены. (`Интерфейс → сервис-шлюз` по
  HTTP/WS — разрешено; это клиентская граница, а не межсервисный вызов.)
- Маршруты **клиентского API** (HTTP/WS для интерфейсов) живут
  **только** на сервисе-шлюзе; версионирование клиентского API выполняется там
  же, а не отдельно в каждом сервисе. Модель — `<methodology-repo>/docs/ARCHITECTURE.md`
  → *Сервис-шлюз*.
- Не изобретать свой конверт события в сервисах — использовать только общий.
- Идемпотентность: подписчик устраняет дубликаты по `event_id`.
