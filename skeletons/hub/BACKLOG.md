# Программный бэклог

Единая очередь «что делать» **по всей программе** (хаб + сервисы + интерфейсы
+ автономные компоненты). Живёт в хабе, не в сервисах. Один источник для всех агентов.

> Скелет. Заполняй под программу. Модель цикла —
> `<methodology-repo>/docs/refs/PIPELINE.md`; роли —
> `<methodology-repo>/docs/refs/ROLES.md`; процедура работы над задачей —
> `<methodology-repo>/docs/guide/30-implement-task.md`.

## Правила

- **Строгая последовательность, программа-wide.** Очередь сверху вниз.
  Параллельные задачи запрещены. Один источник «что делать» — этот файл.
- **Глобальный лок.** Ровно один пункт в работе (`[~]`) на всю программу.
  Пока есть `[~]`, новый `[ ]` **не берётся**. Это блокирует обновления по всей
  системе: агенты работают строго друг за другом, медленно, но без гонок.
- **Без перепрыгивания.** Берётся первый невыполненный `[ ]` (выше него —
  только `[x]`).
- **Definition of Ready.** Захватывается только `[ ] ready` по
  `<methodology-repo>/docs/refs/TASK.md`; агент не додумывает intent.
- **Punt = единственное исключение из «без перепрыгивания».** Если задача
  заблокирована, она получает точный диагностический статус и **съезжает с
  верхушки** вниз —
  чтобы одна застрявшая задача не держала всю программу. Следующий `[ ]`
  берётся. Человек разбирает диагностический статус отдельно.
- **Маркер персистентен через git.** `[ ]` → `[~]` (коммит в хаб:
  `chore(backlog): mark N in-flight`) → `[x]` (`chore(backlog): mark N done`).
  Так лок видим всем агентам/репо — состояние бэклога = состояние гита хаба.
- Человек меняет порядок, добавляет задачи, разбирает диагностические статусы. Агент
  берёт первый `[ ]`, ставит `[~]`, работает, ставит `[x]`.

## Формат задачи

Тег — **целевой репо + модуль** (или `interface`/`stub` + артефакт):

```
### TASK-NNNN. [ ] ready [<сервис>/<модуль>] Краткое название
Репо: <service-repo-url>
Спек: docs/specs/<module>.md, раздел «Что TODO».
Зависит от: M (если есть)
Риск: low | medium | high | critical
Автономность: pr-only | human-before-merge | auto-test-deploy | human-before-production
Владелец intent: <человек>

Цель и наблюдаемое поведение (что, не как).

Ограничения / не входит:
- …

Acceptance criteria:
- AC-1: …

Контракты и данные:
- …

Тесты:
- …

Rollback:
- …
```

Для интерфейса или автономного компонента — тег `[<interface>/…]` / `[<stub>/…]`; работа
приходит изменениями `COMPOSITION`/`CONVENTIONS` хаба и ADR
(`<methodology-repo>/docs/refs/PIPELINE.md`).

## Маркеры

- `[ ]` — не начато.
- `[~]` — in-flight (ровно один на всю программу). Захватывается только через
  merge короткого lock-PR: гейт сверяет актуальный `main`, выбранный первый
  `[ ]` и отсутствие другого `[~]`. Lock-PR проходит через merge queue либо
  обязательное обновление и повторную проверку прямо перед merge; stale-check
  запрещён. Открытый PR или локальный коммит локом не считается.
- `[x]` — сделано.
- `[ ] needs-spec|needs-architecture|needs-access|blocked-external|automation-failed|retry-exhausted` — съехало с верхушки, ждёт устранения точной причины.

### TASK-0001. [ ] ready [<сервис>/<модуль>] Пример задачи
Репо: <service-repo-url>
Спек: docs/specs/EXAMPLE.md, раздел «Что TODO».
Риск: medium
Автономность: auto-test-deploy
Владелец intent: <имя>

Опиши, что нужно сделать. Конкретно: какие типы/функции/поведения появляются.
Укажи ограничения и валидации. Что должно попасть в «Что есть» спеки после
реализации.

Тесты:
- Случай 1: ожидаемое поведение.
- Случай 2: ошибка на невалидном входе.
- Граничный случай.

Rollback:
- Вернуть предыдущий RC; изменение данных отсутствует.
