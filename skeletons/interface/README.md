# <interface>

Репозиторий клиентского приложения. Его страницы, компонентная структура,
доверительная граница и потребляемые gateway endpoints канонически описываются в
`docs/ARCHITECTURE.md`.

## Первичная настройка

Выполните один раз в указанном порядке:

1. Получите у человека назначение интерфейса и целевой gateway из хаба.
2. В `.methodology.yml` закрепите точный тег или commit методологии.
3. Заполните `docs/ARCHITECTURE.md`, включая полный манифест потребляемых
   endpoints; не повторяйте этот реестр в README.
4. В `.env.example` задайте только публичные имена клиентской конфигурации.
5. Зафиксируйте в `AGENTS.md` фактические lint, typecheck, test и build команды.
6. Задайте CI variables `METHODOLOGY_REPOSITORY` и `METHODOLOGY_VERSION`.
7. Укажите лицензию ниже и выполните локальные проверки.

## Карта файлов

| Файл | Назначение | Режим изменения |
|---|---|---|
| `docs/ARCHITECTURE.md` | единственный реестр UI-границ, страниц и gateway endpoints | вместе с интерфейсом |
| `.methodology.yml` | тип репозитория и закреплённая методология | при создании или обновлении методологии |
| `AGENTS.md` | ограничения клиента и точные команды | вместе с toolchain |
| `.env.example` | имена публичной клиентской конфигурации | вместе с конфигурационным контрактом |
| `.github/workflows/verify.yml` | обязательная проверка методологии | вместе с общим CI-шаблоном |
| `.evidence/` | записи выполнения | только автоматизация |

Системный состав и задачи находятся в хабе. Интерфейс обращается только к
зарегистрированному browser-facing gateway и не подключается к брокеру.

## Локальная разработка и сборка

```bash
pnpm dev
pnpm lint && tsc --noEmit
pnpm test
pnpm build
```

Рабочий цикл — `<methodology-repo>/docs/WORKFLOW.md`, поставка статики —
`<methodology-repo>/docs/OPERATIONS.md`.
