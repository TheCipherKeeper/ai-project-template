# ADR 0001 — Записывать архитектурные решения

Date: <YYYY-MM-DD>
Status: Accepted

## Контекст

Нужен способ фиксировать значимые архитектурные решения и причину их принятия.
Без этого агентам и новым разработчикам непонятно, почему система устроена
именно так, и есть риск откатить решение, не зная его обоснования.

## Решение

Завести каталог ADR (`docs/adr/`). Каждое решение — отдельный файл с
однострочной нумерацией (`000N-<slug>.md`). Структура: Контекст / Решение /
Последствия. Старые решения помечаем `Superseded`, не удаляем — история
должна оставаться читаемой.

## Последствия

- Появляется трейл решений: видно, что менялось и почему.
- Расхождение доков или спорное архитектурное изменение → повод для нового ADR.
- Затраты: один файл на значимое решение. Окупается читаемостью для агентов.
