# <service>

Репозиторий одного независимо поставляемого сервиса. Его фактическое назначение,
границы, модули и контракты канонически описываются в `docs/ARCHITECTURE.md`, а
поведение каждого модуля — в `docs/specs/<module>.md`.

## Первичная настройка

Выполните один раз в указанном порядке:

1. Получите у человека имя, назначение и высокоуровневую границу сервиса.
2. Выберите один стек: Python, Go или Rust.
3. Заполните `docs/ARCHITECTURE.md` и создайте краткую спеку для каждого модуля.
   Это каноническая человекочитаемая запись принятой архитектуры.
4. Заполните `.methodology.yml` как машинную проекцию архитектуры: точный ref
   методологии, имя, язык и тот же упорядоченный список модулей.
5. В `AGENTS.md` оставьте одну строку команд выбранного стека.
6. Настройте `.env.example`, `Dockerfile` и локальный `docker-compose.yml`.
7. Задайте CI variables `METHODOLOGY_REPOSITORY` и `METHODOLOGY_VERSION`.
8. Укажите лицензию ниже, создайте структуру кода по архитектурному канону и
   выполните локальные проверки.

## Карта файлов

| Файл | Назначение | Режим изменения |
|---|---|---|
| `docs/ARCHITECTURE.md` | единственный реестр границ, модулей и контрактов сервиса | вместе с устройством сервиса |
| `docs/specs/<module>.md` | поведение, инварианты, ошибки и проверка модуля | вместе с поведением модуля |
| `.methodology.yml` | машинная проекция идентичности и списка модулей из архитектуры | после изменения канонической архитектуры |
| `AGENTS.md` | локальные ограничения и точные команды | при создании или изменении toolchain |
| `.env.example` | имена конфигурационных параметров без секретов | вместе с конфигурационным контрактом |
| `Dockerfile`, `docker-compose.yml` | поставка и локальный запуск с брокером | вместе со способом поставки |
| `.evidence/` | записи выполнения | только автоматизация |

Состав системы, общий envelope и задачи находятся в хабе; сервис не дублирует
`COMPOSITION.md`, `CONVENTIONS.md` или `BACKLOG.md`.

## Локальный запуск

```bash
cp .env.example .env
docker compose up --build
```

Рабочий цикл — `<methodology-repo>/docs/WORKFLOW.md`, эксплуатация —
`<methodology-repo>/docs/OPERATIONS.md`. Стабильный тег создаётся отдельной
задачей.
