# AI Microservice Methodology

[![License: MIT](https://img.shields.io/badge/code-MIT-green)](LICENSE)
[![Docs: CC BY 4.0](https://img.shields.io/badge/docs-CC%20BY%204.0-lightgrey)](LICENSE-DOCS)

**Центральная методология построения микросервисного проекта.** Читается как
пошаговый гайд и служит первоисточником правил для хаба, сервисов, интерфейсов и
stub-таргетов. Это **не код приложения** и **не один микросервис** — это
методология того, **как** строить систему из хаба + N сервисов + M интерфейсов
+ K stub-таргетов. Хаб, сервисы, интерфейсы и stub-таргеты инстанцируются из
`skeletons/` отдельными репо и ссылаются сюда за правилами, **не копируя** их к себе.

## Что здесь

- **`docs/guide/`** — процедуры-плейбуки по фазам (bootstrap → … → release).
  **Пошаговые и модульные:** каждая фаза самодостаточна, читается одна.
- **`docs/refs/`** — авторитетные факты (одна правда, без дублирования).
  Системный уровень: `TOPOLOGY` (структура репозиториев), `COMMUNICATION`
  (общение через брокер, event envelope), `VERIFICATION`
  (verification gate, edge-модель). Per-service: `STACKS` / `LAYOUT` / `MODULE`
  (внутренняя архитектура модуля) / `DEPLOYMENT` / `SPEC`.
- **`skeletons/service/`** — стартовый набор сервис-репо. Копируется → новый репо.
- **`skeletons/interface/`** — стартовый набор interface-репо (React/TS,
  визуализации, потребление presentation-эндпоинтов **gateway-сервиса**).
  Копируется → новый репо.
- **`skeletons/stub/`** — стартовый набор stub-репо (standalone-программа; форма —
  параметр дескриптора: контейнер / CLI / …; без брокера/presentation;
  параметризуется дескриптором; out-of-band-наблюдение collector'ом — режим, не
  свойство). Копируется → новый репо.
- **`skeletons/hub/`** — стартовый набор хаб-репо (`COMPOSITION`, `CONVENTIONS`,
  системный `docker-compose.yml`, `adr/`). Копируется → хаб-репо.
- **`AGENTS.md`** — правила работы над **самой методологией** (ветвление,
  docs-verify, можно/нельзя).
- **`docs/INDEX.md`** — роутер «ситуация → читай».

## Модель

Схема репозиториев и рёбер верификации — `docs/refs/TOPOLOGY.md`.

- **Этот репо** — корень авторитета, не инстанцируется как сервис, не содержит
  кода приложения. Хаб/сервисы/интерфейсы/stub-таргеты ссылаются на `docs/guide/`
  и `docs/refs/` здесь.
- **Хаб-репо** хранит системные контракты (`CONVENTIONS`) и состав программы
  (`COMPOSITION` — сервисы + интерфейсы + stub-таргеты); сервисы потребляют
  текущий `CONVENTIONS`, гейт проверяет сервис против него.
- **Сервис-репо** — клиент брокера (одного на систему: Kafka/Redpanda/NATS),
  один стек (Python/Go/Rust/TS), деплой контейнером; внутри — workspace модулей.
  Один из сервисов — **gateway** (каноническая роль: единственный browser-facing
  surface, экспонирует presentation-эндпоинты для интерфейсов); прочие сервисы
  presentation для интерфейсов не держат (их клиентский край = топики брокера,
  потребляемые gateway).
- **Интерфейс-репо** — React/TS, клиент на границе: зовёт presentation-эндпоинты
  **gateway-сервиса** по HTTP/WS (один URL/CORS/auth); деплой статикой.
- **Stub-репо** — standalone-программа (форма — параметр дескриптора: контейнер /
  CLI / …), не брокер-клиент и не peer, не имеет presentation; параметризуется
  дескриптором из manage; out-of-band-наблюдение collector'ом — режим (включается
  при наличии наблюдаемой поверхности), не свойство; деплой — по форме.

Edge-модель верификации (`методология → хаб → сервисы`,
`методология → интерфейсы`, `методология → stub-таргеты`) —
`docs/refs/VERIFICATION.md`.

## Как пользоваться

- **Построить систему / инстанцировать репо** — `docs/INDEX.md` →
  *Инстанциация → skeletons/* (хаб, сервис, интерфейс, stub) + фазы `guide/00..70`.
- **Разобраться в методологии** — `docs/INDEX.md` (роутер «ситуация → читай»).
- **Разработка в этом репо** — `AGENTS.md` (ветвление, docs-verify, можно/нельзя).

## Разработка (в репозитории методологии)

```bash
git checkout main && git pull && git checkout -b feat/<задача>
# правка guide/refs/skeletons; docs-verify перед коммитом — AGENTS.md
git commit -m "docs(guide): ..." && git push   # PR в main
```

Прямой коммит в `main` запрещён; путь в `main` единственный — feature-ветка +
PR + merge (squash). Это запрет на способ, а не на попадание: влить PR в `main`
— норма и ожидаемое действие. Стабильные версии —
тегами `vX.Y.Z` (`docs/guide/70-release.md`). Подробно — `AGENTS.md`.
Команды запуска сервисов/хаба — в **инстанцированных** репо (там есть код и
`Dockerfile`); в этом репо их нет.
