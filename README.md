**Центральная методология разработки микросервисной системы.** Репозиторий
содержит обязательные правила, пошаговые процедуры, справочные сведения и
исходные шаблоны. Здесь нет прикладного кода отдельного микросервиса.

Методология распространяется на систему, состоящую из хаба, нескольких
сервисов, пользовательских интерфейсов и имитаторов целей. Каждый компонент
создаётся из соответствующего каталога `skeletons/`, размещается в отдельном
репозитории и ссылается на этот репозиторий как на источник правил.

## Что здесь

- **`docs/guide/`** — пошаговые процедуры по этапам разработки. Каждый документ
  самодостаточен и применяется независимо от остальных этапов.
- **`docs/refs/`** — канонические технические сведения без дублирования.
  Системный уровень: `TOPOLOGY` (структура репозиториев), `COMMUNICATION`
  (обмен через брокер и конверт сообщения), `VERIFICATION`
  (контроль соответствия и модель связей). Для отдельного сервиса: `STACKS` / `LAYOUT` / `MODULE`
  (внутренняя архитектура модуля) / `DEPLOYMENT` / `SPEC`.
- **`skeletons/service/`** — исходный шаблон репозитория сервиса.
- **`skeletons/interface/`** — исходный шаблон репозитория пользовательского
  интерфейса на React и TypeScript.
- **`skeletons/stub/`** — исходный шаблон репозитория имитатора цели (`stub`).
  Имитатор выполняется как самостоятельная программа и не подключается к брокеру.
- **`skeletons/hub/`** — исходный шаблон репозитория хаба: состав системы,
  общесистемные соглашения, файл `docker-compose.yml` и каталог ADR.
- **`AGENTS.md`** — правила работы над **самой методологией** (ветвление,
  docs-verify, можно/нельзя).
- **`docs/INDEX.md`** — указатель документов по рабочим ситуациям.

## Модель

Схема репозиториев и проверяемых связей приведена в `docs/refs/TOPOLOGY.md`.

- **Этот репозиторий** является корневым источником правил и не создаётся как
  сервис. Прикладного кода в нём нет.
- **Репозиторий хаба** хранит состав системы (`COMPOSITION`), общесистемные
  соглашения (`CONVENTIONS`), системную конфигурацию запуска и ADR.
- **Репозиторий сервиса** содержит один микросервис на одном языке. Сервис
  подключается к общему брокеру и развёртывается в контейнере. Один сервис может
  выполнять роль **сервиса-шлюза** (`gateway`): только он предоставляет
  клиентский HTTP- и WebSocket-API пользовательским интерфейсам.
- **Репозиторий пользовательского интерфейса** содержит приложение на React и
  TypeScript. Оно обращается только к сервису-шлюзу и развёртывается как набор
  статических файлов.
- **Репозиторий имитатора цели** содержит самостоятельную программу, которая не
  является сервисом системы и не подключается к брокеру. Способ поставки
  определяется формой имитатора: контейнер, программа командной строки или иной
  исполняемый артефакт.

Модель проверки связей (`методология → хаб → сервисы`,
`методология → интерфейсы`, `методология → имитаторы целей`) описана в
`docs/refs/VERIFICATION.md`.

## Как пользоваться

- **Создать репозитории системы** — см. `docs/INDEX.md`, раздел об исходных
  шаблонах, и процедуры `guide/00..70`.
- **Найти нужный документ** — см. `docs/INDEX.md`.
- **Разработка в этом репо** — `AGENTS.md` (ветвление, docs-verify, можно/нельзя).

## Разработка (в репозитории методологии)

```bash
git checkout main && git pull && git checkout -b feat/<задача>
# изменение документов и шаблонов; проверка перед коммитом описана в AGENTS.md
git commit -m "docs(guide): ..." && git push   # PR в main
```

Прямой коммит в `main` запрещён. Изменения вносятся в ветке `feat/<задача>`,
проходят проверку в запросе на слияние и объединяются с `main` одним коммитом
(squash). Стабильные версии отмечаются
тегами `vX.Y.Z` (`docs/guide/70-release.md`). Подробно — `AGENTS.md`.
Команды запуска сервисов/хаба — в **инстанцированных** репо (там есть код и
`Dockerfile`); в этом репо их нет.
